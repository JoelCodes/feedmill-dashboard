<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser timer → router.refresh() | Client-side trigger of RSC re-render; same auth/role gate as initial render; no new attack surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-02-01 | T — Tampering | setInterval cadence (30_000 ms) | accept | Constant is hardcoded; cannot be tampered with at runtime from the browser. A malicious user could only stop the timer (via devtools) — outcome: stale data. Same effect as closing the tab. |
| T-34-02-02 | D — Denial of Service | router.refresh() flooding | accept | router.refresh is internally deduped by Next.js; even if a malicious page mounted N copies of the hook, request frequency is bounded by Next.js's debouncing. v2.0 polling cadence is locked at 30s — well within mill-server capacity (seed data ≤33 rows + low import volume). |
| T-34-02-03 | I — Information Disclosure | Polling re-fetches over the same auth session | accept | Same auth gate as the initial page load. `auth.protect()` runs server-side on each refresh; if the session expires, the user is redirected to `/sign-in` on the next refresh tick. No additional disclosure path. |
| T-34-02-04 | E — Elevation of Privilege | Hook exposes no role-gated behavior | accept | Hook is pure timer + router.refresh; no role check inside. Role enforcement remains in server actions (Phase 33). |
</threat_model>

Scope: Add `router.refresh()` calls in the **success paths** of every mutating transition handler in `TransitionButtons.tsx` AND in `BlockReasonModal.tsx`'s `blockOrder` handler. The conflict path already calls `router.refresh()` (D-14 behavior, lines 67-71 / 100-104 / 147-151 of TransitionButtons.tsx) — preserve that exactly.

Pattern: `useActionState` exposes `[state, formAction, isPending]`. After every successful submit, the `state` updates. Add a success-arm to each component's existing conflict useEffect:

```typescript
useEffect(() => {
  if (state?.ok === true) {
    router.refresh(); // T12 fix — bring latency from 15s polling → ~1s
  } else if (state?.ok === false && state.code === 'conflict') {
    router.refresh(); // D-14 — existing
  }
}, [state, router]);
```

Why this is safe:
- `revalidateTag('production-orders', 'max')` already fires in the action (lines 203, 277-286 of transitions.ts). `router.refresh()` is the client-side counterpart that picks up the invalidated cache.
- The `state?.ok === true` arm cannot fire on the initial render because `useActionState`'s initial state is `null`.
- React 19 deduplicates concurrent `router.refresh()` calls; if the polling tick fires at the same moment, only one round-trip happens.

For `BlockReasonModal.tsx`: the success path already calls `onClose()` (line 56). After this plan it ALSO calls `router.refresh()` BEFORE `onClose()` so the drawer's freshly-fetched data lands before the modal unmounts (preventing a render flash where the drawer briefly shows stale state between modal close and the next polling tick).

Purpose: Cross-tab transition latency goes from ~15s (worst case 30s) to ~1s. Same operator pressing Mixing → Complete now sees the order move columns in real time instead of "did I click that? oh it just appeared".

Output:

- `BlockReasonModal.tsx`: the `useActionState` reducer at lines 51-61 calls `router.refresh()` after a successful `blockOrder`.

useEffect(() => {
  if (state?.ok === true) {
    // T12 fix (2026-05-14): bring cross-tab latency from ~15s polling → ~1s.
    // Server already calls revalidateTag('production-orders', 'max'); this is
    // the client-side counterpart that picks up the invalidated cache.

Pattern for BlockReasonModal:
```typescript
const router = useRouter();

const [state, formAction, isPending] = useActionState<TransitionResult | null, FormData>(
  async () => {
    const result = await blockOrder(orderId, version, reason);
    if (result.ok) {
      // T12 fix: refresh first, THEN close. router.refresh() returns synchronously
      // (it kicks off the RSC fetch async). Closing the modal after kicking off
      // the refresh means the drawer's next paint already has the freshly-fetched
      // Blocked state instead of the stale Pending/Mixing state.

      onClose();
    }
    return result;
  },
  null
);
```

Note: the existing modal already has the `if (result.ok)` branch (lines 54-57). Just insert `router.refresh()` BEFORE the existing setReason + onClose calls.

`onClose()` is called by the parent drawer to flip `modalOpen` back to false; the call ordering inside `if (result.ok)` is: refresh → clear local state → notify parent.

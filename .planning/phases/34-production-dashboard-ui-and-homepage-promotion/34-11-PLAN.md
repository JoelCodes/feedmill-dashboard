<objective>
Close gap T10b (UAT.md: "Drawer opens responsively when a card is clicked — target: <500ms perceived load").

Per the debug session (`.planning/debug/t10b-drawer-slow-load.md`) and the routing decision (UAT.md Gaps table, 2026-05-14): the drawer fetch is already parallelized (`Promise.all` at `src/app/page.tsx:39-43`), the events query already has the correct composite index, and the queries are properly cached. **The bug is in URL-state propagation, not the data layer.**

Root cause: `setQuery({ order: id })` in `ProductionDashboard.tsx:140-144,248,257,266` and `BlockedAlertBand.tsx:28,40` uses nuqs default `shallow: true`. Shallow updates only patch `history.replaceState` and do NOT trigger an App Router RSC re-fetch (verified in `nuqs/dist/impl.app-HXOL9k0k.js:45-55` — `router.replace` only fires when `!options.shallow`). The page RSC (`src/app/page.tsx:39-43`) is the only place `getOrderById` + `getOrderEvents` are called, so when it doesn't re-render the drawer props stay `null` / `[]` and the drawer renders the `order === null` "Order not found" branch. Real drawer data only materializes when `useProductionPolling` (`src/hooks/useProductionPolling.ts:30-36`) fires `router.refresh()` on its 30s interval — wait time after a click is 0–30s, averaging ~15s.

Fix strategy (per UAT routing decision):
1. **Split the `order` key** out of the combined `useQueryStates({ status, q, order })` call into its own `useQueryStates({ order: parseAsString.withDefault('') }, { shallow: false, history: 'push' })`.
2. **Keep `status` and `q` shallow** (preserves the search-debounce performance and pill-toggle responsiveness — these intentionally do not trigger RSC fetches).
3. **Wrap the `order` setter in `React.startTransition`** so the existing `<Suspense fallback={<DrawerSkeleton />}>` boundary at `ProductionDashboard.tsx:275-282` actually renders during the RSC fetch (without startTransition, React 19 treats the update as urgent and skips the fallback).
4. Apply the same split to `BlockedAlertBand.tsx` (the blocked-chip click path) and `ProductionDrawer.tsx` (the drawer close path — closing the drawer is the inverse `?order=` clear and must also propagate to the RSC).

Purpose: Bring drawer perceived load from 0–30s (averaging ~15s) down to ~1s, matching the operator's expectation for a single-click interaction. Preserve the snappy filter/search UX (no needless RSC fetch on every keystroke).

Output:
- `ProductionDashboard.tsx`: split `useQueryStates` into two — `{ status, q }` (shallow default) + `{ order }` (shallow: false, history: 'push'). The `setOrderQuery` is wrapped in `startTransition` everywhere it is called from card clicks.
- `BlockedAlertBand.tsx`: same split (the band already uses its own hook — change its options to `{ shallow: false, history: 'push' }` and wrap the setter in `startTransition`).
- `ProductionDrawer.tsx`: the drawer close path (`handleClose` at line 129) uses the same shallow: false setter so closing the drawer triggers RSC re-fetch (and the drawer cleanly unmounts when `drawerOrder` becomes null).
- Tests confirm:
  - Card click triggers a non-shallow URL update.
  - Pill toggle and search-input keystroke still use shallow updates (no router.replace beyond URL patch).
  - Drawer close also triggers non-shallow update.

From `nuqs` API surface (RESEARCH.md or upstream — verified pattern):
- `useQueryStates(parsers, options?)` — second arg is `{ shallow?: boolean; history?: 'push' | 'replace'; throttleMs?: number; clearOnDefault?: boolean; scroll?: boolean }`.
- `shallow: false` triggers App Router `router.replace`/`router.push` (depending on `history`) which re-runs the RSC. `history: 'push'` is preferred for drawer open/close so the browser back button toggles the drawer (deep-link parity).
- Setter signature when wrapping in startTransition: `React.startTransition(() => setOrderQuery({ order: id }))`. This is the React 19 transition primitive; `useTransition` is not strictly required, but using `startTransition` from the React top-level import is sufficient and avoids a stale-closure pitfall for one-shot setters.

Why startTransition matters here:
Per React 19, state updates that route through the RSC pipeline (router.replace with shallow: false) are "transitions" only if marked. Without `startTransition`, React commits the update synchronously and skips the `<Suspense fallback>`. The user sees the OLD drawer (or no drawer) until the RSC finishes — which feels like a freeze. With `startTransition`, React keeps the old UI visible AND renders the suspense fallback in the new location, so the DrawerSkeleton (`<Suspense fallback={<DrawerSkeleton />}>` at ProductionDashboard.tsx:275-282) appears immediately.

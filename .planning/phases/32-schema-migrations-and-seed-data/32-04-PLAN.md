## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local operator shell → Neon dev DB (over HTTPS) | Operator authenticates via `DATABASE_URL_UNPOOLED` (Neon-issued direct connection string). The connection terminates at Neon; the SQL is fully under operator control. |
| Generated SQL file → drizzle-kit migrate (in-process) | The SQL is operator-reviewed at Task 2 before Task 3 applies it. No SQL is auto-generated and auto-applied without a human-in-the-loop. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-02 | Tampering / Denial of Service | `npx drizzle-kit migrate` accidentally run against PRODUCTION DB | mitigate | Three layers: (1) `DATABASE_URL_UNPOOLED` is the dev URL in `.env.local` only (Vercel env-var provisioning explicitly deferred to Phase 34 first deploy per Phase 31 D-04 / CONTEXT.md OUT_OF_SCOPE). (2) Task 3 is `checkpoint:human-action gate="blocking"` — operator must explicitly confirm "applied" after verifying connection target. (3) Task 2 operator-review gate ensures the SQL is reviewed before any apply. No automated CI runs migrate in Phase 32 — this is documented and deliberate. |
| T-32-04 | Spoofing / Process Abuse | `drizzle-kit push` used instead of `migrate` (creates schema without migration files) | mitigate | Task 1 acceptance criterion explicitly bans the `db:push` script in package.json (verified by grep). Pitfall 5 (32-RESEARCH.md line 774-778) documents recovery steps if push is used by mistake. SC#5 / D-23 ban this command at the project policy level. |
| T-32-06 | Information Disclosure | `.env.local` (containing Neon credentials) committed to git | accept | Already mitigated by Phase 31 — `.env.local` is gitignored (verified by Phase 31-05-SUMMARY.md §"Pre-gate sanity check"). No new env vars added in Phase 32. |
| T-32-08 | Tampering | Generated SQL file committed to git contains malicious / incorrect SQL | mitigate | Task 2 is a human review gate; operator inspects the SQL before Task 3 applies it. Code review (pre-commit) is a second gate. Generated SQL is deterministic from the schema files — repeated `drizzle-kit generate` yields byte-identical output, so any tampering would be detected by re-running generate locally. |

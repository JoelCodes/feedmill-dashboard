## Phase Boundary

Define the four production Drizzle tables (`production_orders`, `order_events`, `import_batches`, `users`) as TypeScript pgTable schemas, generate + apply the initial SQL migration via `drizzle-kit generate` + `drizzle-kit migrate`, and seed the development database with the 33-order Book1.xlsx baseline so dev mirrors the existing `/demo/mill-production` surface. The schema/migration/seed disciplines set in Phase 31 (server-only client, `DATABASE_URL_UNPOOLED` for migrations, no `drizzle-kit push`) are inherited and not relitigated.

**In scope:**
- `src/db/schema.ts` placeholder is replaced by a split directory: `src/db/schema/{orders.ts, events.ts, imports.ts, users.ts, index.ts}`
- `drizzle.config.ts` schema path updated from `./src/db/schema.ts` to `./src/db/schema/index.ts`
- Postgres native enum types `production_state` (Pending/Mixing/Completed/Blocked) and `mill_line` (Premix/Excel/CGM)
- `production_orders` columns: id (uuid PK, defaultRandom), order_number (text unique), customer, product, weight_lbs, delivery_time, state (enum), mill_line (enum), texture_type (nullable text), line_code (nullable text), version (int DEFAULT 1), created_by (text NOT NULL), created_at, updated_at (Drizzle `$onUpdate`)
- `order_events` columns: id (uuid PK), order_id (uuid → production_orders.id, FK with ON DELETE CASCADE), from_state (production_state nullable), to_state (production_state NOT NULL), changed_by (text NOT NULL — clerk_user_id), changed_at (timestamp), note (text nullable)
- `import_batches` columns: id (uuid PK), file_name (text), row_count (int), imported_by (text — clerk_user_id), imported_at (timestamp)
- `users` columns: id (text PK — Clerk's `user_xxx` ID directly, no surrogate), display_name (text), email (text), last_seen_at (timestamp), created_at (timestamp)
- Indexes shipped with migration #1: `idx_orders_state`, `idx_orders_mill_line`, `idx_orders_order_number UNIQUE`, composite `idx_events_order_id_changed_at_desc (order_id, changed_at DESC)`
- One-time JSON snapshot at `src/db/seed-data.json` derived from `src/services/millProduction.ts` (33 orders with millLine + state baked)
- `src/db/seed.ts` reads the JSON, TRUNCATEs `production_orders` / `order_events` / `import_batches` (NEVER `users`), inserts the 33 rows with `created_by = 'system-seed'`
- `package.json` script `db:seed` runs `tsx src/db/seed.ts`; `tsx` added as a dev dependency if not already present
- `src/types/millProduction.ts` rewrite: the hand-written `ProductionOrder` is renamed to `DemoOrder` and `/demo/*` imports are updated; the new canonical `ProductionOrder` is the Drizzle `$inferSelect` exported from `src/db/schema/orders.ts`
- `drizzle/0000_*.sql` (or similarly numbered) migration file checked in, generated by `drizzle-kit generate`
- `drizzle-kit migrate` applied against the Phase 31 Neon dev database
- Verification: `drizzle-kit introspect` (or direct `\d production_orders`) confirms the four tables + enum types + indexes exist; row count after seed is 33

**Out of scope (deferred to later phases):**
- Query functions in `src/db/queries/` → Phase 33 (introduced alongside the actions that consume them)
- Server actions (`src/actions/transitions.ts`, `src/actions/import.ts`) → Phase 33
- The `read-excel-file` 9.0.9 dependency (XLSX parsing) → Phase 33 (Phase 32 seeds from JSON, not XLSX)

<decisions>
## Implementation Decisions

### Schema file layout
- **D-01:** Split `src/db/schema.ts` into a per-table directory: `src/db/schema/{orders.ts, events.ts, imports.ts, users.ts, index.ts}`. `index.ts` is a barrel that re-exports every table + enum + inferred type. Matches `.planning/research/v2.0/ARCHITECTURE.md` Decision 1.
- **D-02:** `drizzle.config.ts` schema path becomes `./src/db/schema/index.ts` (was `./src/db/schema.ts` in Phase 31). The barrel-based path is the Drizzle Kit canonical pattern.
- **D-03:** Drizzle inferred types (`$inferSelect` / `$inferInsert`) are **co-located** in each table file. `src/db/schema/orders.ts` exports `productionOrders` (pgTable) AND `ProductionOrder` / `NewProductionOrder` (inferred types). Single import surface per table. No central `src/db/types.ts`.
- **D-04:** The Drizzle-inferred `ProductionOrder` becomes the **canonical** project-wide type. The existing `src/types/millProduction.ts` `ProductionOrder` (consumed by the demo) is renamed to `DemoOrder`. `/demo/mill-production/page.tsx` and `src/services/millProduction.ts` update their imports. The `MillLine` and `ProductionState` unions in `src/types/millProduction.ts` are deleted; consumers import them as `MillLine = typeof millLineEnum.enumValues[number]` from `src/db/schema/orders.ts` (or the index barrel). Demo continues to function on `DemoOrder` because its shape matches the current TS structure 1:1.
- **D-05:** File naming convention: short names (`orders.ts`, `events.ts`, `imports.ts`, `users.ts`). Exported pgTable variables use the full DB table name in camelCase (`productionOrders`, `orderEvents`, `importBatches`, `users`). This mirrors the variable names used in `.planning/research/v2.0/ARCHITECTURE.md`.

### Column primitives & state encoding
- **D-06:** `production_orders.id` is `uuid primary key default gen_random_uuid()`. ERP-side `order_number` (text) gets a separate `UNIQUE` index for duplicate detection on bulk import (IMPORT-05). Drizzle: `id: uuid('id').primaryKey().defaultRandom()`.
- **D-07:** `state` (Pending/Mixing/Completed/Blocked) and `mill_line` (Premix/Excel/CGM) are **Postgres native ENUM types** — `pgEnum('production_state', [...])` and `pgEnum('mill_line', [...])`. Lives in `src/db/schema/orders.ts` (next to the table that uses them most; re-exported from the index barrel).
- **D-08:** `updated_at` on `production_orders` uses Drizzle `.$onUpdate(() => new Date())`. No DB trigger. Phase 33 server actions don't need to set `updatedAt` manually — Drizzle injects it on every `db.update(...)` call. Direct SQL writes are not in scope, so the JS-level guarantee is sufficient.
- **D-09:** `clerk_user_id` columns exist on three tables, all as `text` with NO foreign key (locked by ROADMAP SC#2):
  - `production_orders.created_by text NOT NULL`
  - `order_events.changed_by text NOT NULL`
  - `users.id text PRIMARY KEY` — stores Clerk's `user_xxx` ID directly; no surrogate ID. DATA-05 says `users` is a lazy-sync display-name cache, never used for auth decisions — using the Clerk ID as the PK makes that explicit.
- **D-10:** `order_events.order_id` has a foreign key to `production_orders.id` with `ON DELETE CASCADE`. Append-only audit log semantics — events live and die with their parent order. `from_state` is nullable (the initial "created" event, if ever produced, has no from-state); `to_state` is NOT NULL.
- **D-11:** `production_orders.version` is `integer NOT NULL DEFAULT 1` (locked by ROADMAP SC#2). The seed inserts rows with version = 1 (the default). Phase 33 actions bump it via `set({ version: sql\`version + 1\` })` inside the optimistic concurrency `UPDATE ... WHERE id = $id AND version = $version`.
- **D-12:** `production_orders.weight_lbs` is `numeric(10, 2)` (NOT `integer`). Mock data uses whole numbers (e.g., 6000) but Book1.xlsx exports decimal quantities (e.g., `6,000.00`); `numeric` handles both. `texture_type` and `line_code` are `text` and NULLABLE (mock data has rows missing texture_type — see April.22.md "Legacy Five Farms Ltd / LAY 9 FREE RANGE OS" with empty texture).
- **D-13:** `production_orders.delivery_time` is `text NOT NULL` (matches the mock service shape `"6:30 AM"`). Not a `time` or `timestamp` — the existing mock + demo treat it as a display string, and the dashboard sorts by mill-line column order, not by delivery_time as a sortable scalar. Phase 35 KPIs may revisit this if scheduled-vs-actual variance lands.
- **D-14:** Timestamps use `timestamp with time zone` (Drizzle `timestamp('column_name', { withTimezone: true })`). Default to `defaultNow()` for `created_at`, `imported_at`, `changed_at`, `last_seen_at`. Phase 31 has not established a precedent — this phase sets the convention for v2.0.

### Seed source & idempotency
- **D-15:** Seed source is a **static JSON snapshot** at `src/db/seed-data.json`. One-time export: a small helper script (could live as a one-shot `scripts/export-seed.ts` or just be performed manually during execution) reads the 33-order array out of `src/services/millProduction.ts`, normalizes the field names to snake_case DB columns, and writes it to JSON. The seed runtime reads the JSON only — no dependency on `src/services/*`. Decoupling rationale: when `src/services/millProduction.ts` evolves for demo polish, the seed baseline stays frozen.
- **D-16:** `npm run db:seed` → `tsx src/db/seed.ts`. `tsx` is added as a devDependency (if not present). The seed script reads `DATABASE_URL_UNPOOLED` for the connection (direct, faster for the 33-row batch). `.env.local` loading mirrors `drizzle.config.ts`'s `dotenv` pattern (line 1: `import { config } from 'dotenv'; config({ path: ... });`).
- **D-17:** Idempotency: seed begins with `TRUNCATE production_orders, order_events, import_batches RESTART IDENTITY CASCADE`. Re-runs give a clean slate. **The `users` table is NEVER truncated** — it's lazily populated by Phase 33 session sync and wiping it would orphan real Clerk-mapped rows.
- **D-18:** Seed scope: 33 `production_orders` rows only. `order_events`, `import_batches`, `users` stay empty after seed. Cleanest day-zero baseline; Phase 34 dev sees a realistic "no audit history yet" state on every seeded order; Phase 33 import work has a clean import_batches surface to populate first.
- **D-19:** All 33 seeded orders use `created_by = 'system-seed'` (literal string). Distinguishable from real Clerk user IDs; no coupling to any specific test user; survives Clerk user rotation. The Phase 34 details panel will display 'system-seed' as the creator for these orders — acceptable for the dev baseline (and an implicit hint to users that this is fixture data, not real production data).

### Indexes in migration #1
- **D-20:** Migration #1 ships four read-path indexes:
  - `idx_orders_state` on `production_orders(state)` — supports the filter pills (PROD-03)
  - `idx_orders_mill_line` on `production_orders(mill_line)` — supports the dominant three-column dashboard query `WHERE mill_line = $1` (PROD-02)
  - `idx_orders_order_number` UNIQUE on `production_orders(order_number)` — duplicate detection for IMPORT-05
  - `idx_events_order_id_changed_at_desc` composite on `order_events(order_id, changed_at DESC)` — supports the canonical "transition history for one order" query in PROD-05 (`WHERE order_id = $1 ORDER BY changed_at DESC`)
- **D-21:** No substring/typo-tolerant search indexes in Phase 32. `pg_trgm` extension + GIN indexes on `customer` and `product` are deferred to Phase 34 if measurement under realistic load shows ILIKE is too slow. At dev scale (33 rows) and realistic prod scale (low hundreds), sequential scan inside the mill_line-filtered subset is fast enough.
- **D-22:** No composite `(mill_line, state)` index. The two single-column B-tree indexes already let Postgres do bitmap-AND for `WHERE mill_line = $1 AND state = ANY($2)`. A composite is only justified if EXPLAIN shows the bitmap-AND plan is slow under load — measure first.

### Migration discipline (carried from Phase 31)
- **D-23:** `drizzle-kit push` is **banned** for any v2.0 work after this phase's initial schema. The `drizzle/` directory holds committed SQL migration files; the dev workflow is `drizzle-kit generate` → review → commit → `drizzle-kit migrate`. CI verification of "re-runnable from scratch" (ROADMAP SC#3) is a manual operator action documented in the phase plan: `drizzle-kit drop` against a scratch Neon branch, then `drizzle-kit migrate` from the committed files.
- **D-24:** Migration filename naming uses drizzle-kit's default sequence-based scheme (`0000_*.sql`). Phase 32 produces a single migration; whether to split table creation into multiple sequenced migrations is a planner call but is not required.

### Claude's Discretion
- Drizzle `relations()` declarations — not strictly required for this phase since Phase 32 produces no queries. Planner may omit relations entirely and let Phase 33 add them when the first `db.query.productionOrders.findMany({ with: { events: true } })` lands. Recommended: omit in Phase 32 to keep the schema files lean.
- Whether the JSON export of `src/services/millProduction.ts` mock data is generated by a checked-in `scripts/export-seed.ts` script (idempotent, re-runnable) or as a one-time hand transformation. Recommended: write a small `scripts/export-seed.ts` so the relationship between the demo data and seed data is documented in code and re-derivable when demo data evolves.
- Test approach: a Jest test that asserts seed JSON row count = 33 and every row has required NOT NULL columns is cheap insurance. Whether to assert against the live DB after seed (integration test) is left to planner — it requires CI access to a Neon test branch which has not been provisioned.
- Whether to drop `src/types/millProduction.ts` entirely after D-04 or keep it as a re-export shim for the demo. Planner decides based on how many demo imports point at it.

</decisions>

### Build-time risks Phase 32 must surface
- **Drizzle Kit ESM/CJS quirks with tsx**: `tsx` runs TS directly via esbuild but Drizzle Kit and `drizzle-orm` mix ESM/CJS. The Phase 31 `drizzle.config.ts` uses `defineConfig` with `import` syntax + `__dirname` polyfill (verify in the file). The seed script should mirror this pattern to avoid `__dirname is not defined` or `Cannot use import statement outside a module` errors.
- **Enum migrations and naming**: Postgres enum types are GLOBAL within a schema namespace. `pgEnum('production_state', ...)` creates a type that survives `DROP TABLE production_orders`. If a future migration changes the enum values, `ALTER TYPE production_state ADD VALUE ...` is required (drizzle-kit handles this in generate but the diff is visible — review carefully).
- **`gen_random_uuid()` availability**: Neon enables pgcrypto by default, so `gen_random_uuid()` works without an explicit extension migration. Drizzle's `defaultRandom()` compiles to `DEFAULT gen_random_uuid()`. Phase 32 should not need a `CREATE EXTENSION` migration; verify by reading the first generated SQL file.
- **`numeric(10, 2)` vs JS `number`**: Drizzle infers `numeric` columns as TS `string` by default to preserve precision. Phase 33 queries will need to parse `weight_lbs` as a number for UI display. Note for downstream; not a Phase 32 blocker.
- **FK CASCADE on order_events**: `ON DELETE CASCADE` means deleting an order destroys its audit history. This is intentional for dev (seed re-runs truncate orders, events follow). For production, no delete pathway exists in scope (orders are state-machine-driven, never deleted) — but the cascade is a footgun if a future phase adds a "delete order" capability without auditing the cascade behavior. Note for the phase plan.

<specifics>
## Specific Ideas

- **The seed is for development parity with `/demo`, not for production data**. The seeded 33 orders are the same fictionalized data that drives `/demo/mill-production` today (Chick Magnet Farms, Fowl Play Poultry, etc.). Real production data starts arriving via Phase 33 bulk import (Book1.xlsx). Seeded rows can be distinguished by `created_by = 'system-seed'` if anyone audits the table.
- **`mill_line` in seed = the same hand-coded distribution the demo already uses** (~11 Premix, ~11 Excel, ~11 CGM). The seed does NOT derive mill_line from XLSX columns because Book1.xlsx itself has no mill_line column — it groups orders by species (Poultry / Cattle / Hog). Phase 33 IMPORT-02 will need a rule to derive mill_line from imported XLSX rows; that rule is OUT of Phase 32 scope.
- **`order_number` in seed is taken verbatim from the mock's `orderNumber` field** (e.g., `"ORD-255154"`). The Book1.xlsx `Document Number` column (e.g., `255154`) is the un-prefixed version; the demo prefixes with `ORD-`. Phase 33 IMPORT-02 must decide whether to prefix imported document numbers — flagged as a Phase 33 question, not a Phase 32 decision.
- **`drizzle-kit drop` + re-migrate verification** for ROADMAP SC#3 ("re-runnable from scratch") is an operator action, documented in the phase plan as a manual verification step. No automated CI test for this in Phase 32 — adding one requires a CI-accessible Neon branch which is not provisioned.
- **`users` table is a forward-looking placeholder for Phase 33 session sync.** Phase 32 only DEFINES it; it stays empty. The schema choice of `id text PRIMARY KEY` (Clerk's `user_xxx` literal) means the sync code is a one-line upsert.

</specifics>

Close the single open UAT gap for Phase 32: `/demo/mill-production` triggers a LightningCSS parse error
because Tailwind v4's Oxide scanner picks up the literal token `text-[var(--text-&ast;)]` (rendered
with a literal asterisk in the offending files) from five `.planning/**/*.md` files and compiles a
malformed CSS rule. This is the THIRD recurrence of the same class of bug
(Phase 27 -> Phase 31-05 -> Phase 32). The goal is to make it the LAST recurrence by enforcing
a runnable gate that catches the next re-introduction.

Apply a THREE-layer fix atomically in one plan:
  - Layer 1 (durable defusal): replace the asterisk inside the token with the HTML/Markdown entity
    `&ast;` in every tracked institutional-memory file that quotes the bug — including the five
    originally-diagnosed `.planning/**/*.md` files, the debug session file, AND this PLAN.md itself.
    Also clean up the transient `.claude/worktrees/agent-*/` directory that mirrors six copies of the
    literal (it is a stale `gsd-debugger` worktree that did not auto-cleanup).
  - Layer 2 (structural backstop): replace the broken `@source not "../../.planning/**/*"` directive
    in `src/app/globals.css` with `source(none)` + an explicit positive `@source` for `src/`. This is
    the most explicit Tailwind v4 construct — it cannot silently no-op the way `@source not` with a
    glob did.
  - Layer 3 (recurrence-prevention enforcement): add `src/__tests__/no-bad-tailwind-literals.test.ts`
    — a Jest test that scans `.planning/**/*.md` and `src/**/*` for the dangerous pattern and fails
    on match. Jest is already invoked by `npm test` (the existing CI gate), so this enforcement runs
    on every test run with zero new tooling. The next engineer who writes the literal in a new
    SUMMARY, REVIEW, or debug session will see a failing test instead of a silent landmine.

Purpose: prevent a landmine that has now bitten three phases, and add an active gate so a fourth
recurrence is caught at test-time rather than at UAT-time. The diagnosis records at
`.planning/debug/css-text-var-text-star-parse-fail.md` and the Phase 32 UAT gap-notes are KEPT IN
PLACE as institutional memory — only the parseable token is neutered, not the prose.

Output: `src/app/globals.css` with a working source scope; five `.planning/**/*.md` files + the debug
file + this PLAN.md with defused tokens; the `.claude/worktrees/agent-a3690773d1b688bc2` directory
removed; a new Jest test enforcing the rule; a clean `npm run build` + clean `/demo/mill-production`
dev load; a green `npm test` run that now includes the enforcement gate.

NOTE on entity rendering: the `&ast;` defusal works because Tailwind v4's Oxide scanner reads raw
bytes from the source file — it tokenizes on the literal characters and the `&` byte is not a valid
Tailwind class character, so no utility is generated. Some Markdown renderers MAY resolve `&ast;`
back to `*` at display time. That is an acceptable trade-off: the binding constraint is the scanner
(byte-level), not the rendered view. This is documented here so future readers do not "fix" the
defusal by reverting to a literal asterisk.

- Phase 27 originally had `@source not "../../.planning"` (directory form). Empirically it did NOT
  recurse into .planning/&ast;&ast;/&ast;.md for this Tailwind v4 + Oxide combo (see 27/deferred-items.md

- Phase 31-05 commit 3fb044b "fixed" it by switching to `@source not "../../.planning/&ast;&ast;/&ast;"` (glob
  form). Tailwind v4 docs + community discussion (#18550) confirm: glob form is UNSUPPORTED and silently
  no-ops. Phase 31-05 went green only because the same commit also scrubbed 18-UI-REVIEW.md — the scrub
  did the work, the glob change broke the directive.
- Phase 32 markdown writing reintroduced the literal token (UAT gap-notes quote the bug verbatim), so
  the scanner picks it up again. Build breaks.
- The first revision of THIS plan (32-07) itself contained 9 copies of the literal in its action prose
  and rationale — a self-invalidating artifact. This revision uses the entity form throughout to
  defuse the plan file itself.

Why Layer 1 alone is insufficient: any future SUMMARY / REVIEW / debug session that quotes the bug
will reintroduce a parseable token. The `@source not` directive MUST be repaired so Tailwind never
scans .planning/ at all.

Why Layer 2 alone is insufficient: even with a working directive, the literal sits as a latent
landmine in five tracked files. One config drift, one new tooling version, and it bites again.

Why Layer 3 is the new ingredient: prose declarations don't enforce anything. The plan's goal is
"the LAST recurrence" but Layers 1+2 are passive — they don't catch the next engineer who writes
the literal in a new file. A Jest test that grep-scans the tree turns the rule into a runnable
contract: regression -> failing test -> blocked commit / blocked CI.

Why option (b) `source(none)` + explicit positive `@source` over option (a) directory form:
  - Phase 27 empirically observed directory form did NOT recurse for this codebase. Repeating the
    same construct trusts an unverified assumption.
  - Option (c) (.gitignore .planning/) untracks the planning tree — unacceptable (project memory).
  - Option (b) is the most explicit Tailwind v4 construct: `source(none)` opts out of auto-detection
    entirely, then positive `@source` directives whitelist exactly what should be scanned. There is
    no silent no-op failure mode — if the syntax is wrong, the build fails loudly with no utilities
    found.

Why Mechanism 3 (Jest test) over Mechanism 1 (npm script) or Mechanism 2 (husky pre-commit):
  - Jest is already wired into `npm test` and into the project's TDD discipline (Wave-0 contracts).
    Adding one more test file is the cheapest, most-idiomatic mechanism.
  - The default Jest config in `jest.config.ts` does NOT ignore `src/__tests__/` — it only ignores
    `node_modules/` and `e2e/` (verified). The new test file is picked up automatically.
  - Husky is not installed (verified: no `.husky/` directory, no `husky` in devDependencies). Pulling
    it in just for this would be invasive.
  - A standalone `npm run` script is fine but adds a new entry point developers must remember.
    A test that fails on `npm test` is loud, automatic, and aligns with existing habits.
</background>

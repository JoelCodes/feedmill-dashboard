  <action>
Lines 380-388 have clickable div rows without keyboard support:
- click-events-have-key-events: needs onKeyDown handler
- no-static-element-interactions: needs role="row" or use proper table semantics

NOTE: The table already has keyboard navigation via the parent container's onKeyDown (lines 330-331), but individual rows also need to be clickable.

Add role and keyboard handler to each row div (around line 380):

```typescript
<div
  data-order-id={order.id}
  onClick={() => handleSelectOrder(order.id)}
  onKeyDown={(e) => {
    if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      handleSelectOrder(order.id);
    }
  }}
  role="row"
  tabIndex={-1}
  className={`flex cursor-pointer items-center py-3 transition-colors
    ${validSelectedId === order.id
      ? 'bg-[color-mix(in_srgb,var(--primary)_10%,transparent)]'
      : 'hover:bg-[var(--bg-page)]'
    }`}
>
```

Using role="row" with tabIndex={-1} allows the row to receive focus programmatically (for keyboard navigation) but not via tab (since the container handles tab focus).

Also add role="rowgroup" to the rows container and role="table" to the main container if needed for full ARIA compliance.

<task type="auto">
  <name>Task 2: Fix setState-in-effect in orders page</name>
  <files>src/app/orders/page.tsx</files>
  <action>
Lines 14-20 have a useEffect that calls setSelectedOrderId synchronously, triggering react-hooks/set-state-in-effect warning:

```typescript
useEffect(() => {
  const urlSelected = searchParams.get("selected");
  if (urlSelected && urlSelected !== selectedOrderId) {
    setSelectedOrderId(urlSelected);
  }
}, [searchParams, selectedOrderId]);
```

The purpose is to sync URL param to state. Better approaches:

Option 1: Initialize state from URL param (remove effect entirely):
```typescript
const searchParams = useSearchParams();
const initialSelected = searchParams.get("selected");
const [selectedOrderId, setSelectedOrderId] = useState<string | null>(initialSelected);
```

Wait - this is already done on line 12. The useEffect is trying to handle URL changes after initial load (e.g., navigating from timeline).

Option 2: Use useSyncExternalStore or remove the dependency on selectedOrderId:
```typescript
useEffect(() => {
  const urlSelected = searchParams.get("selected");
  if (urlSelected) {
    setSelectedOrderId(urlSelected);
  }
}, [searchParams]); // Remove selectedOrderId from deps
```

This still triggers on URL change but avoids the cascading render warning. The comparison `urlSelected !== selectedOrderId` was to prevent unnecessary updates, but React already optimizes same-value setState calls.

Actually, the simplest fix is to remove the conditional entirely - if URL has a selected param, just set it. React's setState is idempotent for same values.

Or suppress the rule for this specific case with a comment explaining why:
```typescript
// eslint-disable-next-line react-hooks/set-state-in-effect -- Syncing URL param to state on navigation
```

Use Option 2 (remove selectedOrderId from deps) as the cleanest fix.

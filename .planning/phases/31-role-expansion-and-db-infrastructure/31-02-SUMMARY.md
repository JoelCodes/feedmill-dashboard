## Decisions Made

- **Exact-version pinning via `--save-exact`.** Per RESEARCH line 114, Phase 31 establishes a known-good baseline. Floating semver (`^`) could allow a future `npm install` (e.g., adding a new dev tool) to silently upgrade drizzle-orm and break the Phase 32 migration generator. Locked exact.
- **Drizzle-kit generate verification deferred from Task 2 Step 3 to after Task 3.** The Plan's Task 2 Step 3 instructs running `drizzle-kit generate` immediately after creating src/db/{index,schema}.ts, but the command requires a drizzle.config.ts that doesn't yet exist at that point. Ran it after Task 3 instead — confirmed Assumption A1 (0 tables, no errors). The fallback pgTable was not needed.
- **Cleaned up drizzle-kit's generated `./drizzle/meta/` directory** after the verification ran. The empty `_journal.json` (0 entries) is a drizzle-kit artifact that Phase 32 will regenerate alongside its first real migration. Keeping the empty journal in Phase 31's commits would have committed a build artifact NOT in the plan's `files_modified` set and would have created merge friction when Phase 32 runs its first generate.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Jest 30 deprecated `--testPathPattern` (singular) → `--testPathPatterns` (plural)**
- **Found during:** Task 2 (test execution after writing RED test)
- **Issue:** The plan's automated verification command and Task 2 Step 1 instruction both used `--testPathPattern='db/__tests__/index'`. Jest 30 (already installed at jest@30.3.0 per package.json line 45) prints `Option "testPathPattern" was replaced by "--testPathPatterns"` and exits with an error before running tests.
- **Fix:** Used `--testPathPatterns='db/__tests__/index'` throughout. Same regex, same semantics, only the flag name changed.
- **Files modified:** None (CLI invocation only — no source code change).
- **Verification:** `npm test -- --testPathPatterns='db/__tests__/index' --watchAll=false` runs all 5 assertions correctly (RED then GREEN).
- **Committed in:** N/A (no file change; documented here so the orchestrator does not silently re-issue the deprecated flag).

**2. [Rule 1 — Bug] JSDoc in src/db/index.ts contained the literal substring `'use client'`**
- **Found during:** Task 2 GREEN (running tests after creating index.ts)
- **Issue:** My first draft of the JSDoc explanation for `import 'server-only';` referenced the literal string `'use client'` (in the phrase "transitive import from a `'use client'` file"). The 5th test assertion checks `expect(content).not.toContain("'use client'")` as a defense against accidentally putting a `'use client'` pragma in the file. The test (correctly) flagged the JSDoc substring.
- **Fix:** Rewrote the JSDoc phrase to "transitive import from a client component or Edge-runtime bundle" — semantically equivalent, no literal `'use client'` substring.
- **Files modified:** `src/db/index.ts` (JSDoc comment only).
- **Verification:** 5/5 tests pass; `grep -c "'use client'" src/db/index.ts` returns 0.
- **Committed in:** `5082108` (rolled into the GREEN commit before staging).

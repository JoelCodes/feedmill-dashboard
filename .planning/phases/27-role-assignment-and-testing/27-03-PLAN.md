- `interface CustomJwtSessionClaims { metadata: { role?: Role } }` where `Role = 'demo' | 'admin' | 'user'`
- JWT template JSON (verbatim, D-05): `{"metadata": {"role": "{{user.public_metadata.role}}"}}`
- This shape selects only the `role` field (not the whole `public_metadata` object) — keeps the token under Clerk's 1.2 KB custom-claim ceiling per RESEARCH Pitfall 3.

Test user matrix (per D-12, D-13):
| Email | publicMetadata.role | Used by |
|-------|---------------------|---------|
| e2e-demo+clerk_test@example.com | "demo" | demo-user Playwright project (D-11 #1, #3) |

The `+clerk_test` mailbox suffix is Clerk's "test mode" pattern that disables real email delivery during verification — combines D-12's `e2e-{role}` prefix with Clerk's safe-mailbox marker (RESEARCH Open Question 2 recommendation).

1. Update the JWT template in Clerk Dashboard (Sessions → Customize session token).
2. Create the three users (Users → Add user) with the emails above.
3. For each user, edit `publicMetadata` and set `role` (Users → click user → Metadata → Edit publicMetadata).
4. If existing dev sessions are open, sign out and sign back in so the new template is picked up.
5. Put the six `E2E_*` keys with real values into `.env.local` (gitignored).

<objective>
Define the canonical Zod validator for one row of the Book1.xlsx-format import file. This schema is consumed by both `previewImportAction` (plan 33-05) and `commitImportAction` (plan 33-06) — placing it in its own dedicated file (`src/actions/import-schema.ts`) avoids forcing the `'use server'`-directive co-location problem in `import.ts` (per Pitfall 2, only files marked `'use server'` can be imported by client components as server actions; a co-located schema exported from the same file would be ambiguous).

Purpose: A single failing-fast validation surface for XLSX rows. Required fields (Document Number, Customer, Product, Weight, Mill Line, Early Delivery Date, Formula Type) reject on absence with field-named error issues per D-15. Optional fields (Texture Type, Line Code) accept null AND undefined per RESEARCH.md §Pitfall 8 (read-excel-file returns `undefined` for absent cells; `.nullable()` rejects undefined; `.nullish()` accepts both). Weight rejects on non-positive values per D-15.

This plan is TDD-pure: pure unit logic, no DB, no `next/cache`, no `requireRole`. Maximally testable in isolation.

Output: One new source file (`src/actions/import-schema.ts`), one new test file. No production code in plan 33-05/33-06 yet — those depend on this plan landing first.
</objective>

import { z } from 'zod';

export const productionOrderImportSchema = z.object({
  orderNumber:  z.string().min(1, 'Document Number is required'),
  customer:     z.string().min(1, 'Customer is required'),
  product:      z.string().min(1, 'Product is required'),
  weightLbs:    z.number().positive('Weight must be positive'),
  deliveryTime: z.string().min(1, 'Early Delivery Date is required'),
  formulaType:  z.string().min(1, 'Formula Type is required'),
  millLine:     z.enum(['Premix', 'Excel', 'CGM']).default('Premix'),
  textureType:  z.string().nullish(),  // null OR undefined OK (D-15 + Pitfall 8)
  lineCode:     z.string().nullish(),  // null OR undefined OK
});

export type ProductionOrderImportRow = z.infer<typeof productionOrderImportSchema>;
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Untrusted XLSX content -> Zod validator | The XLSX file is operator-uploaded content; row-shape can be anything. Zod's `safeParse` returns a discriminated `{success, data} | {success, error}` union — there is no exception path that could leak parser state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-Input | Tampering / Input Validation | XLSX row payload | mitigate | `productionOrderImportSchema.safeParse(row)` rejects rows with missing required fields, non-positive weight, or invalid `millLine` enum values. Per-field error messages let plans 33-05/33-06 surface row-level errors to the operator (IMPORT-04 requirement). |
| T-33-Coercion | Tampering | weightLbs number-to-string boundary | mitigate | `weightLbs` is validated as `z.number().positive()`; the number-to-string coercion is deferred to the action body so the validator boundary is clean. The Task 3 JSDoc + the inline `CR-01` reference prevent silent regressions. |
| T-33-NullVsUndefined | Tampering / Schema Drift | textureType, lineCode | mitigate | `.nullish()` accepts BOTH `null` and `undefined`, matching the read-excel-file output shape for absent cells. Pitfall 8 is locked at test level (Test 11 + Test 12). |
</threat_model>

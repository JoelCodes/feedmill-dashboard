<objective>
Close VERIFICATION.md **GAP-01** (Concurrent transition race / SC#2 — exact-once locked message) by adding a scripted live-DB harness that exercises the actual Postgres-level optimistic-concurrency race that unit tests cannot reach.

The unit tests in `src/actions/__tests__/transitions.test.ts` mock `@/db` and simulate the conflict path by overriding `returning.mockResolvedValueOnce([])`. That proves the action's code path handles a zero-rows response correctly, but it does NOT prove that Postgres' `UPDATE … WHERE id = $id AND version = 1 RETURNING id` actually returns zero rows for exactly one of two concurrent calls. GAP-01 is the integration-level gap: prove the Postgres-level atomicity claim against a real Neon dev DB.

Harness shape:

1. Seed ONE production_orders row: `state: 'Pending'`, `version: 1`, `created_by: 'harness-race-test'` (sentinel for cleanup).
2. Fire TWO `db.update(productionOrders).set({ state: 'Mixing', version: sql\`version + 1\` }).where(and(eq(id, seedId), eq(version, 1))).returning({ id: productionOrders.id })` calls in parallel via `Promise.all`. Each call replicates the action's optimistic-update statement.
3. Categorize results: exactly one should return `[{ id: seedId }]` (winner), the other should return `[]` (loser).
4. For the loser, construct the discriminated-union response `{ ok: false, code: 'conflict' as const, message: CONFLICT_MESSAGE }` where `CONFLICT_MESSAGE` is the locked string `'Order was modified by another user. Please refresh.'`. Assert `message === expectedLockedString` byte-for-byte (no whitespace drift, no curly-quote slip).
5. SELECT the final row state. Assert `state === 'Mixing'`, `version === 2`.
6. Cleanup via sentinel-keyed DELETE.

Why NOT invoke the action directly: same reason as plan 33-07 — the action's `requireRole('mill_operator')` calls Clerk's `auth()` which throws without request context. The gap is about driver-level / DB-level race outcome, not about the auth wrapper. The harness replicates the optimistic UPDATE verbatim (same Drizzle expression, same `version + 1` sql literal, same `.returning()` shape) so the DB-level guarantee is exercised end-to-end.

Scope discipline:
- This plan adds ONE script and ONE `npm run` entry.
- It does NOT modify `src/actions/transitions.ts` or any file owned by 33-01..33-06.
- If the harness reveals a real bug (e.g., both UPDATEs succeed, or both return zero rows), flag and STOP — do not patch transitions.ts. The bug would indicate that Postgres/Drizzle/Neon HTTP is not providing the atomicity the spec assumes — a significant finding worthy of operator review before continuing.

Output: `scripts/test-concurrent-transition.ts` + `package.json` script entry. Operator runs `npm run test:concurrent-race`, observes exit 0, and updates `33-HUMAN-UAT.md` Test #1 to `passed (harness, <date>)`.

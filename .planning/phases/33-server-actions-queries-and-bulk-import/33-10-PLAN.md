Close VERIFICATION.md **GAP-04** (previewImportAction and commitImportAction handle clean files without crashing) by patching a TypeError-on-clean-file bug surfaced by the 33-07 harness on its first live-DB run (2026-05-14).

**The bug** lives at `src/actions/import.ts:136` inside the private `parseAndValidate` helper:

```
for (const pe of parserErrors) {  // ← throws TypeError when parserErrors is undefined
```

**Why it slipped through:** `read-excel-file` v9.0.9 returns `errors: undefined` (not `[]`) when zero parser errors occur — confirmed by `node_modules/read-excel-file/types/parseSheetData/parseSheetData.d.ts` lines 3-13, which define two overloads: `ParseSheetDataResultSuccess` returns `errors: undefined`, `ParseSheetDataResultError` returns `errors: Error[]`. Existing unit tests in `import-preview.test.ts` (Tests 3-15) and `import-commit.test.ts` (Tests 1-26) ALWAYS mock the return value with `errors: []`, so the undefined branch was never exercised. The bug only surfaces against the real library — discovered when 33-07's `scripts/test-xlsx-import.ts` parsed a clean Book1.xlsx fixture and the action threw `TypeError: parserErrors is not iterable`.

**Production impact:** ANY clean XLSX upload (zero parser errors) crashes both `previewImportAction` and `commitImportAction` before reaching Zod validation. The crash is caught by the outer `try`/`catch` and surfaced as a generic `{ ok: false, code: 'server', message: 'Failed to parse XLSX file.' }`, masking the root cause. Every successful import path is blocked.

**Why this is one fix, not two:** Even though the gap-closure inputs describe "two occurrences" (previewImportAction line 136 + commitImportAction's re-parse path), `grep -nF parseAndValidate src/actions/import.ts` confirms a single shared helper:
- `parseAndValidate` is defined at line 117.
- `previewImportAction` calls it at line 348.
- `commitImportAction` calls it at line 523.
- The `for (const pe of parserErrors)` loop appears EXACTLY ONCE (line 136), inside the shared helper.

Both callers share the same code path. Fixing the single loop closes both observable bugs. The fix is:
1. Update the XlsxFn type cast (lines 123-126) so `errors` is `Array<...> | undefined`.
2. Replace `for (const pe of parserErrors)` (line 136) with `for (const pe of parserErrors ?? [])`.
3. Add one regression test per action mocking `errors: undefined`.

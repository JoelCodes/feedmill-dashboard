<objective>
Ship the `previewImportAction` server action — the read-only first half of the XLSX import flow (D-05). It:

1. Enforces `requireRole('mill_operator')` (AUTH-02).
2. Validates client-side-bypassed oversize files server-side (IMPORT-07 layer 2).
3. Parses the uploaded `.xlsx` file via `read-excel-file/node` 9.0.9 (IMPORT-02; banned: xlsx/SheetJS per STATE.md + CVE-2023-30533).
4. Validates each row with `productionOrderImportSchema` (plan 33-03) — collects per-row errors without rejecting the whole file (IMPORT-04 partial-import semantics start at preview).
5. Detects intra-file duplicates (same `orderNumber` appearing twice in the file — RESEARCH.md Pitfall 7).
6. Detects file-vs-DB duplicates via a single SELECT against `productionOrders.orderNumber` for the full set of parsed orderNumbers.
7. Returns a full preview payload `{ summary, rows }` per D-06 — the operator-facing preview table in Phase 34 will render this verbatim.

Purpose: Closes IMPORT-01 (upload via FormData), IMPORT-02 (server-side parse with read-excel-file), IMPORT-03 (preview shows row count, total weight, duplicates), and IMPORT-07 (server-side size validation — the framework body-size limit from plan 33-01 is layer 3, this action is layer 2).

This plan splits from plan 33-06 (commit) because the two actions have different mutation semantics (preview never writes; commit writes per-row + import_batches row). They share the parse + Zod validate + duplicate detection logic, which we extract into a private helper inside `import.ts` and exercise from both actions (commit re-parses per D-05, so the helper runs twice — that is the intended design).

Output: One source file (`src/actions/import.ts`) with three exports (the action, the constant, the result type) plus an internal helper. One test file. No production code in plan 33-06 yet — that depends on this plan landing first because both actions live in the same file.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser (operator upload) -> server action | The uploaded `.xlsx` file is operator-supplied content. File size, content structure, row count, cell values are all untrusted. |
| Server action -> read-excel-file/node parser | The parser is the only code that interprets the binary XLSX format. It must be the SAFE parser (read-excel-file 9.0.9), NOT xlsx/SheetJS (banned per STATE.md + CVE-2023-30533). |
| Server action -> Postgres SELECT (duplicate lookup) | A single batch SELECT with `inArray(...)` — Drizzle parameterizes; no SQL injection vector. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

**The bug** (root cause): `src/actions/import.ts:133` imports the **default export** `readXlsxFile` and calls it with `{ schema: xlsxSchema }`. Per `node_modules/read-excel-file/node/index.d.ts`:

- `export default function readXlsxFile(input, options?): Promise<Sheet<ParsedNumber>[]>` — returns a **2D Sheet array**. No `schema` parameter. The schema option is silently dropped at runtime.
- `export function readSheet(input, options: OptionsWithSchema): Promise<ParseSheetDataResult>` — the schema-aware function. Returns `{ objects, errors }` (discriminated union).

Per `node_modules/read-excel-file/types/parseSheetData/parseSheetData.d.ts`:
- `ParseSheetDataResultSuccess`: `{ objects: Object[]; errors: undefined }`
- `ParseSheetDataResultError`: `{ objects: undefined; errors: Error[] }`

**Symptom timeline:**
1. Plans 33-05/33-06 introduced `readXlsxFile(buf, { schema })` with a fictional `{ rows, errors }` destructure cast. Unit tests mocked the fictional shape. Tests passed because mocks match mocks.
2. GAP-03 (33-07) added a live-DB harness. First run crashed at `for (const pe of parserErrors)` because the real v9.x library returns `errors: undefined` (not `[]`) when `schema` is dropped.
3. GAP-04 (33-10) added `parserErrors ?? []` guard. Harness now reached `rawRows.length` and crashed again — `rawRows` is also undefined (the real return is a 2D `Sheet[]`, destructured as `.rows` → undefined).
4. GAP-05 (this plan) is the API migration: switch from `readXlsxFile` (default export, no schema) to `readSheet` (named export, takes schema, returns `{ objects, errors }`).

**Why this is one migration, not many:** Same as 33-10's "one helper, two callers" reasoning — `parseAndValidate` in `src/actions/import.ts` is the sole call site in both `previewImportAction` and `commitImportAction`. Migrating that one call site closes both action surfaces. The harness at `scripts/test-xlsx-import.ts` duplicates the same pattern (33-07 documented this verbatim) and must be migrated in lockstep.

**GAP-04's `?? []` guard stays.** The v9.x discriminated union means `parserErrors` is genuinely `undefined` on the success branch. The guard from 33-10 is correct — this plan only renames the producer (`readXlsxFile` → `readSheet`) and the source key (`rows` → `objects`), then adds a parallel guard for `rawRows` (now also undefined on the error branch).

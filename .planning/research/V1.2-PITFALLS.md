# Pitfalls Research: v1.2 Customers Page

**Domain:** Adding Customer Management, Activity Feeds, and Bin Monitoring to Existing Operations Dashboard
**Researched:** 2026-05-01
**Confidence:** MEDIUM

## Critical Pitfalls

### Pitfall 1: Offset-Based Pagination Breaking at Scale

**What goes wrong:**
Customer lists and order history pages use offset-based pagination (`LIMIT 20 OFFSET 10000`), which requires the database to scan and count all preceding records. For customers with thousands of orders or large customer databases, retrieval time grows linearly with offset depth, causing pages deep in the list to timeout or take 10+ seconds to load.

**Why it happens:**
Developers implement the simplest pagination pattern first (offset/limit) without considering that some customers will have 1,000+ orders spanning multiple years. The performance problem only appears after real data accumulates.

**How to avoid:**
- Use cursor-based (keyset) pagination with `WHERE id > :cursor_id` which leverages indexes efficiently
- For customer order history specifically: paginate by date ranges first, then by order ID within date ranges
- Add database indexes on commonly queried fields (customer_id, order_date, order_id)
- For v1.2: Mock data should include at least one customer with 100+ orders to expose this early

**Warning signs:**
- Page load times increase as users navigate to later pages
- Database query logs show OFFSET values above 1000
- Users complain about slow "View All Orders" functionality
- No database indexes defined on pagination fields

**Phase to address:**
Phase 2 (Customer Detail Infrastructure) — implement cursor-based pagination for order history before building UI

### Pitfall 2: Activity Timeline Feed Aggregation Wasting Database Reads

**What goes wrong:**
The unified activity timeline aggregates events from multiple sources (orders, deliveries, bin alerts). Without proper architecture, the system fetches ALL events from each source, then sorts and discards most of them in-memory. With thousands of events per customer, this creates massive database load and slow page rendering.

**Why it happens:**
You don't know which items will make up each page of the unified feed until you request them all. The naive approach is to fetch the most recent N items from each source, merge them in-memory, sort by timestamp, and take the first page. This works with 10 events but fails with 10,000.

**How to avoid:**
- Pre-aggregate timeline events into a single `customer_activity` table with indexed timestamps
- Use database-level sorting: `SELECT * FROM customer_activity WHERE customer_id = ? ORDER BY timestamp DESC LIMIT 20`
- Implement write-time aggregation: when an order/delivery/alert is created, insert a corresponding timeline entry
- Store only the summary data in the timeline table (type, timestamp, summary text) with foreign keys to full records
- For v1.2 mock implementation: create a single `getCustomerActivity()` service that returns pre-merged events

**Warning signs:**
- Activity timeline API makes 3+ separate database/service calls per page load
- Timeline rendering time increases with customer history length
- Database query logs show full table scans on orders/deliveries/alerts tables
- Memory usage spikes when loading customer detail pages

**Phase to address:**
Phase 1 (Design & Data Modeling) — define the timeline data structure to support efficient aggregation from the start

### Pitfall 3: Infinite Scroll Memory Leaks in Activity Feeds

**What goes wrong:**
Users scroll through hundreds of timeline events. Every event stays mounted in the DOM with images, charts, and event listeners. After 10 minutes of scrolling, browser memory climbs from 50MB to 500MB+ and the tab crashes or becomes unresponsive.

**Why it happens:**
Developers implement infinite scroll by appending items to a list without unmounting off-screen items. React keeps all components in memory even when scrolled out of view. Event listeners for scroll detection aren't cleaned up on component unmount.

**How to avoid:**
- Use virtualization/windowing libraries (react-window, react-virtuoso) that unmount off-screen items
- For activity timelines specifically: keep maximum 50 items in DOM, unmount items >500px above/below viewport
- Clean up scroll event listeners in useEffect cleanup: `return () => removeEventListener()`
- Use Intersection Observer instead of scroll listeners for better performance
- Lazy load images only when items enter viewport
- For v1.2: Include 100+ timeline items in mock data to test this early

**Warning signs:**
- Browser DevTools shows increasing DOM node count while scrolling
- Memory profiler shows growing heap size without plateauing
- React DevTools shows hundreds of mounted components
- Console warnings about "Can't perform state update on unmounted component"
- Users report sluggish scrolling after viewing multiple customer timelines

**Phase to address:**
Phase 3 (Activity Timeline Component) — implement virtualization from the start, not as a later optimization

### Pitfall 4: Stale Customer Data Across Pages

**What goes wrong:**
User views customer list, sees "3 pending orders". Clicks into customer detail, sees "2 pending orders". Updates order status, returns to customer list, still shows "3 pending orders". Data inconsistency destroys user trust and causes incorrect business decisions.

**Why it happens:**
Each page fetches its own data independently without cache invalidation. Customer list caches aggregated counts. Customer detail fetches fresh order data. When order status changes on detail page, the list page cache isn't updated. This is exacerbated when using separate mock data services for each page.

**How to avoid:**
- Use single source of truth for customer data (React Query, SWR, or Redux with normalized cache)
- Implement cache invalidation: when order status updates, invalidate customer list cache entries
- For v1.2 mock services: use a shared singleton data store, not separate arrays per service
- Add `lastUpdated` timestamps to cached data and display staleness warnings after 5 minutes
- Consider optimistic updates: update UI immediately, sync to server in background
- For cross-page navigation: pass cache hints via URL params or shared state

**Warning signs:**
- Different pages show different counts for the same data
- Users refresh pages constantly to see latest data
- Bug reports about "wrong numbers" on customer list
- Each page component has its own `useState` for shared data
- No cache invalidation logic in data mutation functions

**Phase to address:**
Phase 2 (Customer Detail Infrastructure) — establish shared data layer before building multiple pages

### Pitfall 5: Bin Alert Threshold False Positives

**What goes wrong:**
Bin monitoring shows "CRITICAL: Bin 4A at 95% capacity" alert. User checks physical bin, it's actually at 60%. Team starts ignoring all bin alerts as unreliable. Real critical alert is missed, farm runs out of feed, delivery delayed, customer loses trust.

**Why it happens:**
Sensor calibration drift over time, environmental factors (temperature, humidity, dust buildup), improper threshold configuration (90% alert for a bin that's always 85% full), lack of threshold tuning based on real usage patterns, displaying raw sensor data without validation or smoothing.

**How to avoid:**
- Implement multi-level thresholds: Info (75%), Warning (85%), Critical (95%)
- Add threshold hysteresis: don't clear warning until drops to 80% (not immediately at 84%)
- Display data quality indicators: last calibration date, sensor health status
- Smooth sensor readings: use 5-minute rolling average, not instant values
- Provide "dismiss false alert" feedback mechanism to tune thresholds
- For v1.2: Mock different alert scenarios including threshold-crossing edge cases
- Document expected alert frequency (e.g., "expect 1-2 warnings per week per bin")

**Warning signs:**
- Users report bins showing wrong levels consistently
- Alert fatigue: notifications are ignored or disabled
- No documented calibration schedule for sensors
- Thresholds are hardcoded without customer-specific tuning
- Alert history shows rapid on/off toggling (threshold thrashing)
- No validation against actual delivery records

**Phase to address:**
Phase 4 (Bin Monitoring Component) — implement threshold configuration and validation logic with the initial visualization

### Pitfall 6: Timeline Time Zone Display Confusion

**What goes wrong:**
Order placed at "2:00 PM" in customer's local time (CST) displays as "8:00 PM" in timeline because it's stored/displayed in UTC. Delivery scheduled for "6:00 AM" shows as "yesterday 11:00 PM". Users can't correlate timeline events with their actual experience or printed delivery schedules.

**Why it happens:**
Backend stores timestamps in UTC (correct), but frontend displays them in UTC without conversion. Or worse: mixed display where some events show local time, others show UTC. Developers in PST test with data in PST, don't notice the issue. Problem only appears when users in different time zones view the same timeline.

**How to avoid:**
- Store all timestamps in UTC on backend (continue current practice)
- Always display in user's local time zone on frontend using browser's time zone
- For feed mill operations: likely all users in same facility time zone, so use mill's configured time zone
- Show relative times for recent events: "2 hours ago" instead of "2:00 PM"
- Display time zone label on absolute times: "2:00 PM CST"
- For v1.2 mock data: include events with timestamps across midnight boundaries to test this
- Use date-fns or day.js with consistent time zone handling throughout app

**Warning signs:**
- Timeline shows events in wrong chronological order after DST changes
- Users report times "being off by X hours"
- Inconsistent time display across different pages
- Event times don't match printed delivery schedules or order confirmations
- Hardcoded UTC offset calculations instead of time zone library usage

**Phase to address:**
Phase 3 (Activity Timeline Component) — establish time display conventions before building timeline UI

### Pitfall 7: Next.js Dynamic Route Prefetch Overload

**What goes wrong:**
Customer list page displays 25 customers. Next.js automatically prefetches all 25 customer detail pages (`/customers/[id]`) simultaneously when user hovers over or scrolls past links. Browser makes 25+ simultaneous HTTP requests, overwhelming the server or API rate limits. Page becomes sluggish, mobile data usage spikes.

**Why it happens:**
Next.js Link component prefetches by default for performance. With list pages containing dozens of dynamic route links, this creates excessive network traffic. Developers don't notice on fast dev machines with localhost, but production users on slower connections experience degraded performance.

**How to avoid:**
- Disable automatic prefetch on list pages: `<Link prefetch={false}>`
- Implement hover-based prefetch only: prefetch when user hovers for 200ms
- For mobile: disable prefetch entirely, rely on server-side rendering speed
- Add loading.tsx to customer detail route for instant navigation feedback
- Use selective prefetch: only prefetch first 5 visible items
- For v1.2: Create Link wrapper component with controlled prefetch behavior

**Warning signs:**
- Network tab shows dozens of prefetch requests on page load
- API rate limit errors in console
- Slow initial page load on customer list
- Mobile users report high data usage
- Prefetch requests for customers user never clicks

**Phase to address:**
Phase 1 (Design & Data Modeling) — define navigation patterns and prefetch strategy before implementing list page

### Pitfall 8: Inconsistent Design Patterns Across Pages

**What goes wrong:**
Orders page uses multi-select filter pills with blue active state. Customer page uses single-select dropdown with green active state. Mill production uses card-based layout. Customers uses table layout for similar data. Each page feels like a different app, users get confused, development velocity slows as every new page requires custom components.

**Why it happens:**
Different developers build different pages without shared component library. Rush to ship features leads to "whatever works" implementations. Existing design system (filter pills, status badges) not fully documented or discoverable. New features (customer management) added without reviewing existing patterns.

**How to avoid:**
- Audit existing components BEFORE designing new pages (StatusBadge, FilterPill already exist)
- Extend existing patterns: use FilterPill for customer status filtering, matching orders/mill-production
- Document design decisions: why FilterPill uses toggle behavior, when to use pills vs dropdowns
- Create pattern library: screenshot examples of filter UI, status displays, data tables
- For v1.2: explicitly map customer page components to existing equivalents
  - Customer status indicators → use StatusBadge with customer-specific colors
  - Customer filtering → use FilterPill with same interaction model
  - Customer list → extend existing table patterns from orders page

**Warning signs:**
- Design discussions about "what UI pattern should we use?" without checking existing app
- Copy-paste code instead of importing shared components
- CSS utilities duplicated across files (`bg-blue-500` in one place, `bg-indigo-600` in another)
- PRs adding new components that duplicate existing functionality
- Inconsistent spacing, colors, typography across pages

**Phase to address:**
Phase 1 (Design & Data Modeling) — audit existing patterns, establish reuse plan before any implementation

### Pitfall 9: Mock-to-Real API Type Mismatches

**What goes wrong:**
Mock customer service returns `{ ordersCount: 5, ordersWithChanges: 2 }`. Real API returns `{ order_count: 5, orders_with_changes: 2 }` (snake_case). Or mock always includes `binData` field, but real API only includes it for customers with bins. App crashes in production with "Cannot read property 'level' of undefined".

**Why it happens:**
Mock data is created by frontend developers guessing API shape. Real API is built later by backend team following different naming conventions. TypeScript types are defined based on mock data, not API contract. Optional fields in real API are always present in mock data, hiding undefined handling bugs.

**How to avoid:**
- Define TypeScript interfaces FIRST based on expected real API contract
- Use snake_case to camelCase transformation layer (create once, reuse everywhere)
- Mark optional fields as optional in types: `binData?: BinData`
- Include "incomplete" mock data: some customers without bins, some without orders
- Test with `undefined` field values explicitly: `binData: undefined` not just omission
- For v1.2: Mock data service should match real BinSentry API patterns if documented
- Use Zod or similar for runtime validation: fail fast if API returns unexpected shape

**Warning signs:**
- TypeScript types all use required fields (`field: Type`) without optionals
- No runtime validation on API responses
- Mock data always includes every possible field
- Different casing conventions (camelCase vs snake_case) across services
- No documented API contract or OpenAPI spec to validate against
- Integration tests only run against mocks, not real API

**Phase to address:**
Phase 2 (Customer Detail Infrastructure) — define proper types and validation before building UI that depends on them

### Pitfall 10: Mixing Charting Libraries for Bin Visualization

**What goes wrong:**
Orders page uses Chart.js for timeline. Someone adds Recharts for bin level bars. Another developer needs a gauge chart, adds ECharts. Now the app bundles 3 charting libraries (200KB+ each), inconsistent theming (different colors, fonts, tooltips), and complex maintenance (learning 3 APIs, updating 3 dependencies).

**Why it happens:**
Each new visualization chooses the library with the best example for that specific chart type. No one considers cumulative bundle size or consistency. Chart.js has great line charts, Recharts has nice bar charts, ECharts has beautiful gauges. Each choice seems optimal in isolation.

**How to avoid:**
- Choose ONE charting library at project start, stick with it
- For v1.2 specifically: bin level bars are simple enough for CSS-only solution (no library needed)
  - Use div with width percentage: `<div style={{ width: `${level}%` }} />`
  - Add color based on threshold: green <75%, yellow 75-90%, red >90%
  - Reserve charting library for complex visualizations only (multi-series timelines)
- If existing app uses Chart.js for timeline, use Chart.js for bin charts too
- Document: "We use Chart.js for all data visualization" in architecture docs
- Review existing bundle: what charting libraries already included?

**Warning signs:**
- Multiple chart libraries in package.json
- Bundle size increases significantly with each new chart
- Charts have inconsistent visual style (colors, fonts, animations)
- "Which library should I use for X chart?" questions in PRs
- Different tooltip/legend behavior across different charts in same app

**Phase to address:**
Phase 1 (Design & Data Modeling) — decide charting approach before designing bin visualization

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Separate mock data per page | Faster independent page development | Data inconsistency, hard to sync state across pages | Never (always use shared data layer) |
| Hardcoded alert thresholds | Ships quickly, no UI needed | False positives, requires code changes to tune | Only for v1.2 MVP, must add config UI in v1.3 |
| No pagination on order history | Simpler implementation, works with small datasets | Page breaks when customers have 100+ orders | Only if guaranteed <50 orders per customer |
| Fetch all activity types separately | Simple service layer, follows existing patterns | Slow performance, wasted DB reads | Acceptable for v1.2 mock, must aggregate for real API |
| Client-side timeline merging | No backend changes needed | Memory usage, slow rendering, scale issues | Only with <100 total events across all sources |
| No virtualization on timeline | Simpler component code, no new dependencies | Memory leaks on long sessions | Never (implement from start, hard to add later) |
| Copy-paste existing components | Faster than refactoring to shared component | Divergence, inconsistent updates, bloated codebase | Only as temporary spike, must refactor before merge |
| UTC display without conversion | Works for developers in single timezone | Confusion for users in different timezones | Never (time handling must be correct from start) |

## Performance Traps

Patterns that work at small scale but fail as usage grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Fetching all customers on list load | Slow initial render, increasing over time | Implement pagination with limit 25 per page | >100 customers |
| Loading full order history on customer detail | Page timeout, high database load | Paginate order history, load 20 most recent | Customers with >50 orders |
| Rendering all timeline events in DOM | Memory usage increases while scrolling, eventual crash | Use virtualization (react-window/react-virtuoso) | >100 events per customer |
| Separate API calls per activity type | Timeline load takes 3-5 seconds | Pre-aggregate into single customer_activity query | >500 total events |
| No database indexes on customer_id | Order history queries get slower over time | Add index on orders.customer_id before launch | >1000 total orders |
| Client-side sorting of timeline events | UI freezes on large datasets | Sort on database/server with ORDER BY timestamp DESC | >500 timeline items |
| Eager loading all bin data for all customers | Slow customer list rendering | Lazy load bin data only on customer detail page | >50 customers with bins |
| No query result caching | Every page navigation refetches same data | Implement React Query or SWR with 5-minute cache | Any scale, immediate benefit |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Exposing bin sensor IDs in client-side code | External actors could access or manipulate sensor data | Use opaque customer IDs, resolve to sensor IDs server-side |
| Displaying unfiltered alert messages | Sensor error messages might expose internal system details | Sanitize and map sensor codes to user-friendly messages |
| Customer search without authorization checks | Users could search for and view competitors' data | Implement customer-level access control on search API |
| Unvalidated customer ID in URL parameters | Access to arbitrary customer detail pages via URL manipulation | Validate customer ID against allowed customer list for current user |
| Client-side bin threshold configuration | Malicious users could disable critical alerts via DevTools | Store thresholds server-side, provide read-only UI in v1.2 |

## UX Pitfalls

Common user experience mistakes in this domain.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Showing bin level as precise percentage (73.2847%) | Implies false precision, sensor accuracy is ±5% | Round to nearest 5%: 70%, 75%, 80% |
| Timeline events without context | "Order updated" - updated from what to what? | Show diff: "Order status: Producing → Ready" |
| No loading state on customer search | Users think app froze, type more characters | Debounce search 300ms, show "Searching..." indicator |
| Alert notifications without next action | "Bin 4A at 95%" - okay, now what? | Include actionable info: "Schedule delivery within 2 days" |
| Absolute timestamps for recent events | "2026-05-01 14:32:17" for event 30 minutes ago | Use relative times: "30 minutes ago" for events <24h old |
| No empty state on customer with no orders | Blank page, users think it's broken | Show: "No orders yet" with call-to-action to create order |
| Customer list without recent activity indicator | Can't tell active customers from dormant accounts | Add "Last order: 2 days ago" or "Inactive 6 months" |
| Bin visualization without target fill level | Users don't know if 60% is good or bad | Show target range: "Target: 40-60%" with visual indicator |

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Offset pagination performance issues | MEDIUM | 1. Add cursor-based pagination API endpoint (keep old for compatibility) 2. Update UI to use cursor pagination 3. Deprecate offset pagination after migration |
| Timeline aggregation performance issues | HIGH | 1. Create customer_activity aggregation table 2. Backfill existing events 3. Add write-time aggregation triggers 4. Update frontend to use new endpoint |
| Infinite scroll memory leaks | MEDIUM | 1. Add react-virtuoso dependency 2. Wrap timeline in Virtuoso component 3. Add windowing configuration 4. Test with 500+ item timeline |
| Stale data across pages | LOW | 1. Add React Query or SWR 2. Wrap data fetching in query hooks 3. Add cache invalidation on mutations 4. Migrate pages incrementally |
| False bin alerts | LOW | 1. Add threshold configuration UI 2. Implement rolling average smoothing 3. Add alert acknowledgment/feedback mechanism 4. Retrain thresholds based on feedback |
| Timezone display bugs | LOW | 1. Audit all timestamp displays 2. Add consistent date formatting utility 3. Replace manual formatting with utility 4. Add timezone indicator to all absolute times |
| Next.js prefetch overload | LOW | 1. Create Link wrapper component with prefetch={false} 2. Replace all list page Links with wrapper 3. Add hover-based prefetch selectively |
| Inconsistent design patterns | HIGH | 1. Audit all pages for pattern usage 2. Create shared component library 3. Refactor pages to use shared components 4. Document patterns in Storybook |
| Mock-to-real API type mismatches | MEDIUM | 1. Add runtime validation with Zod 2. Update TypeScript types to match real API 3. Add integration tests against real API 4. Fix undefined handling bugs |
| Multiple charting libraries | MEDIUM | 1. Choose primary charting library 2. Reimplement charts in single library 3. Remove unused chart dependencies 4. Document charting standards |

## Key Insights from Research

### The Proven Pattern Already Exists

Your OrdersTable component implements filter architecture correctly. **v1.1 is replication + adaptation, not invention.**

✓ useMemo for derived state (not useEffect)
✓ Asymmetrical filter counts (self-referential exclusion)
✓ Debounced search input
✓ Multiple filter dimensions coordinated correctly

**Recommendation:** Copy OrdersTable filter pattern, adapt field names and dimensions for mill production view.

### Design System is Already In Place

Globals.css and Tailwind token system is strong. **v1.1 should use it exclusively, never bypass.**

✓ Color tokens defined (--primary, --text-primary, --divider, etc.)
✓ Spacing scale available (gap-1.5, px-2.5, py-2, etc.)
✓ Border radius tokens defined (rounded-xl = 15px)

**Recommendation:** Create design token extraction spreadsheet for .pen file, verify all implementation uses tokens (ESLint rule: no hardcoded colors).

### Mock Data Needs Validation

Service layer is correct pattern (async with simulated delay), but **type safety depends on runtime validation.**

✓ ProductionOrder interface defined
✓ Mock service follows async pattern
✓ Simulated network delay included

**Recommendation:** Add runtime validation in service layer to catch structure mismatches before they propagate. Use factory function for consistent test data.

### Phase Order Matters

**Dependencies:**
- Phase 1 (Filters) → Phase 2 (Data) → Phase 3 (Polish)

**Why:** Filter logic correctness must be verified before wiring data. Data integration must be solid before design polish. Can't go backward easily.

**Recommendation:** Test thoroughly before phase transitions. Don't start Phase 2 until Phase 1 count logic verified. Don't start Phase 3 until Phase 2 data service working.

## Critical Questions for Stakeholders

Answers inform implementation details:

1. **Filter Persistence:** Should filters stick across page refresh?
   - If YES: Requires URL query params (more complex, deferred to v1.2?)
   - If NO: Component state is fine, current approach works

2. **Mill Line Filtering:** Should users filter by mill line?
   - If YES: Adds a filter dimension, affects count matrix
   - If NO: Mill lines always shown, simpler counting logic

3. **Additional Filters:** Texture type? Customer? Or just state + mill?
   - Affects filter count calculation complexity
   - Plan architecture to scale

4. **Real API Timeline:** When will API be available?
   - DTO mapper preparation depends on this
   - May need to adjust mock data structure

5. **Data Volume:** How many orders per day/week?
   - Affects performance optimization decisions
   - Current mock: 12 orders (fast), real: could be 500+ (slower)

Phase 1 (Filters):
- [ ] Filter count matrix tested (all combinations work correctly)
- [ ] Empty state implemented (clear UX when 0 results)
- [ ] Design tokens used exclusively (no hardcoded colors/spacing)

- [ ] Mock data validated (type-safe, runtime checks)
- [ ] Filter logic works with service data
- [ ] Loading/error states implemented
- [ ] DTO mapper prepared for real API

Phase 3 (Design Polish):
- [ ] Pixel-perfect visual match to .pen file
- [ ] All interactive states tested (default, hover, active, disabled)

- [ ] Cross-browser testing passed

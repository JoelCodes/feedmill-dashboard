# Pitfalls: Adding Filters, Design Polish & Data Services to Existing Dashboards

**Project:** CGM Dashboard — Mill Production View v1.1
**Domain:** React dashboard feature additions (filter pills + design refinement + mock services to existing working view)
**Researched:** 2026-04-28
**Scope:** Common mistakes when extending existing components with new interactive features and replicating proven patterns

---

## Executive Summary

v1.1 adds three layers of complexity to an existing working system:
1. **Filter Pills** — Status filtering with count badges (proven pattern from OrdersTable)
2. **Design Polish** — Pixel-matching design tokens, spacing, and colors to approved .pen file
3. **Data-Driven Service** — Mock service with realistic data structure from Book1.xlsx

Each layer introduces failure modes that are harder to fix retroactively than to prevent upfront. The good news: your `OrdersTable` component provides a template for filter architecture. The risk: blindly copying filter logic without understanding the full decision tree for count calculation.

## Critical Pitfalls

### Pitfall 1: Filter Count Logic Breaks When Copied

**What goes wrong:**
Your `OrdersTable` has sophisticated count logic across three `useMemo` blocks:
- `statusCounts` — filtered by search + hasChanges, but NOT by status (to show counts for all statuses)
- `hasChangesCount` — filtered by status + search, but NOT by hasChanges itself
- Filter results — filtered by all three dimensions

When copying this pattern to mill production view with different filter dimensions (production state, mill line, texture type), the count calculation becomes incorrect. Pill shows "5 Completed" but filtering shows 3 orders.

**Why it happens:**
Filter logic is interdependent and asymmetrical. A self-referential filter (e.g., "How many orders are Complete?" shouldn't filter out non-Complete orders first). This is counterintuitive. Copy-paste without understanding the full matrix leads to bugs.

Your code correctly excludes the filter dimension being counted from the count calculation, but explaining this pattern is easy to skip, and future maintainers won't know why.

**Consequences:**
- Pill counts don't match filtered results (looks broken)
- Adding a new filter dimension (e.g., customer) requires rethinking all count calculations
- Users distrust the interface ("it says 5 but shows 3, what's wrong?")
- Phase 3 polish reveals the bug when testing interaction

**Prevention:**
1. **Document the count calculation matrix before coding:**
   ```
   statusCounts calculation:
     Input data: all orders
     Filter: search ✓ | hasChanges ✓ | status ✗ (self-referential)
     Output: { Completed: X, Mixing: Y, ... }

   hasChangesCount calculation:
     Input data: all orders
     Filter: search ✓ | status ✓ | hasChanges ✗ (self-referential)
     Output: number

   Filtered results for rendering:
     Input data: all orders
     Filter: search ✓ | hasChanges ✓ | status ✓
     Output: visible order[]
   ```

2. **Create a test matrix** covering all combinations:
   ```typescript
   const testCases = [
     { statuses: [], search: '', hasChanges: false, name: 'all-off' },
     { statuses: ['Completed'], search: '', hasChanges: false, name: 'status-only' },
     { statuses: [], search: 'Westbridge', hasChanges: false, name: 'search-only' },
     { statuses: ['Completed'], search: 'Westbridge', hasChanges: true, name: 'all-on' },
   ];
   ```

3. **Write an assertion helper:**
   ```typescript
   function validateFilterCounts(
     orders: ProductionOrder[],
     statusCounts: Record<ProductionState, number>,
     hasChangesCount: number,
     filteredResults: ProductionOrder[]
   ) {
     const orderedByStateInCounts = orders.filter(o => statusCounts[o.state] > 0);
     const allCountsSum = Object.values(statusCounts).reduce((a, b) => a + b, 0);

     // Assertion: sum of counts should not exceed total orders
     if (allCountsSum > orders.length) {
       throw new Error('Count logic broken: counts sum > total');
     }

     // Assertion: any visible order should be "counted" in some dimension
     filteredResults.forEach(order => {
       if (statusCounts[order.state] === 0) {
         throw new Error(`Order ${order.id} visible but not counted in status`);
       }
     });
   }
   ```

4. **Test after every filter change:**
   - Apply status filter → verify count for other statuses shrinks appropriately
   - Apply search → verify hasChanges count adjusts
   - Toggle hasChanges → verify status counts unchanged
   - Clear all filters → pill counts should sum to total orders

**Detection:**
- Visual mismatch: Pill shows "5 Pending" but clicking it shows 3 rows
- Count formula differs between this component and OrdersTable (red flag)
- Adding a new filter causes count logic to recompute incorrectly

**When to address:**
Phase 1 (Filter Pills) — test count logic before moving to design polish. Incorrect counts persist visually and damage user trust.

### Pitfall 2: Design Tokens Not Used (Hardcoded Colors & Spacing)

**What goes wrong:**
Design says "use primary color (#4fd1c5) for active pills," but implementation hardcodes the color: `style={{ backgroundColor: '#4fd1c5' }}`. Later, design changes primary to #48d5d1. Component doesn't update automatically because color is hardcoded, not a token reference.

Your project has excellent token system in globals.css (`--primary`, `--divider`, `--shadow-sm`, etc.), but nothing prevents bypassing it with hardcoded values.

**Why it happens:**
Hardcoding feels faster than learning token names. Designer uses eyedropper on Figma, developer copies hex directly. Looks good for the moment, causes maintenance nightmare later.

**Consequences:**
- Visual inconsistency (some components use tokens, others hardcoded)
- Maintenance nightmare (design change requires hunting through codebase)
- No single source of truth for colors/spacing
- Bundle size bloat (duplicate color/spacing values)

**Prevention:**
1. **Use Tailwind/CSS tokens exclusively** (never hardcoded hex or px values):
   ```typescript
   // RIGHT
   <button className="bg-primary text-white px-2.5 py-2 rounded-xl gap-1.5">
   <div className="gap-3 p-5.25 shadow-[0_3.5px_5px_rgba(0,0,0,0.02)]">

   // WRONG
   <button style={{ backgroundColor: '#4fd1c5', padding: '8px 10px' }}>
   <div style={{ gap: '12px', padding: '20px', boxShadow: '0 2px 8px rgba(0,0,0,0.05)' }}>
   ```

2. **Create a design token reference document:**
   ```markdown
   ## Token Mapping

   | Design Name | Token Reference | Value |
   |------------|-----------------|-------|
   | Primary | bg-primary | #4fd1c5 |
   | Text | text-text-primary | #2d3748 |
   | Spacing (gap) | gap-3 | 12px |
   | Border Radius | rounded-xl | 15px |
   | Shadow | shadow-[custom] | 0 3.5px 5px rgba(0,0,0,0.02) |
   ```

3. **When adding custom shadow values** (not in Tailwind defaults):
   ```typescript
   // In globals.css or tailwind.config.ts
   const config = {
     theme: {
       extend: {
         boxShadow: {
           'card': 'var(--shadow-card)',
           'sm': 'var(--shadow-sm)',
         },
       },
     },
   };

   // In component:
   <div className="shadow-card">
   ```

4. **Code review checklist:**
   - No `#hex` colors in JSX
   - No `style={{}}` attributes with color/spacing values
   - All spacing uses Tailwind utilities (px-, gap-, m-, p-)
   - All colors use class names (bg-, text-, border-)

**Detection:**
- Grep for `style={{` or `#[0-9a-f]{6}` in components
- Design changes color, multiple files need updating
- Spacing inconsistent between components (some use py-2 vs others use padding: '8px')

**When to address:**
Phase 3 (Design Polish) — implement pixel-matching with token system from start to avoid refactor.

### Pitfall 3: Mock Data Structure Doesn't Match Component Filter Expectations

**What goes wrong:**
You create a mock service from Book1.xlsx with field names that don't match the component filter logic. Component filters by `order.state === "Pending"` (exact match), but mock returns `order.status: "pending"` (case mismatch). Filter breaks silently—no TypeScript error, just 0 results.

Or: Component expects `deliveryTime: "8:30 AM"` (formatted string), but service returns `deliveryDate: 1743148200` (Unix timestamp). Both represent time, but in different formats. Filter logic breaks.

**Why it happens:**
Mock data created independently from component code. Service layer design and component assumptions evolve separately. Small mismatches accumulate:
- Type mismatch (ProductionState values don't match actual mock data)
- Format mismatch (string vs number, 12-hour vs 24-hour time)
- Missing fields (component assumes field X, service doesn't provide it)

Your mill production types are correct (`ProductionState = "Completed" | "Mixing" | "Blocked" | "Pending"`), but mock data in `getProductionOrders()` must exactly match this contract.

**Consequences:**
- Component gets `undefined` for expected fields, breaks silently
- Filter logic doesn't match any records (0 results even with data)
- Swapping mock for real API breaks components because shapes differ
- Requires refactor before shipping, sometimes major component rewrites

**Prevention:**
1. **Define types first, mock second** (type-driven development):
   ```typescript
   // types/millProduction.ts — source of truth
   export type ProductionState = "Completed" | "Mixing" | "Blocked" | "Pending";
   export type MillLine = "Premix" | "Excel" | "CGM";

   export interface ProductionOrder {
     id: string;
     orderNumber: string;
     customer: string;
     product: string;
     weightLbs: number;
     deliveryTime: string; // HH:MM AM/PM format
     state: ProductionState; // Must match type exactly
     millLine: MillLine;
   }
   ```

2. **Create a mock factory function** to ensure all fields exist:
   ```typescript
   // services/millProduction.ts
   function createMockOrder(overrides: Partial<ProductionOrder> = {}): ProductionOrder {
     return {
       id: "1",
       orderNumber: "ORD-255154",
       customer: "Westbridge Farm",
       product: "BROILER BRD 16% OS",
       weightLbs: 6000,
       deliveryTime: "8:30 AM",
       state: "Completed", // Must be one of the union types
       millLine: "Premix",
       ...overrides,
     };
   }

   const mockOrders: ProductionOrder[] = [
     createMockOrder(),
     createMockOrder({ orderNumber: "ORD-255421", state: "Mixing" }),
     // ...
   ];
   ```

3. **Validate at runtime** (catch mismatches before shipping):
   ```typescript
   const VALID_STATES: ProductionState[] = ["Completed", "Mixing", "Blocked", "Pending"];

   function validateProductionOrder(data: unknown): ProductionOrder {
     if (typeof data !== 'object' || !data) throw new Error('Invalid order');
     const order = data as Record<string, unknown>;

     if (!VALID_STATES.includes(order.state as ProductionState)) {
       throw new Error(`Invalid state: ${order.state}. Expected one of: ${VALID_STATES.join(', ')}`);
     }

     if (typeof order.deliveryTime !== 'string') {
       throw new Error(`Invalid deliveryTime: expected string, got ${typeof order.deliveryTime}`);
     }

     return order as ProductionOrder;
   }

   export async function getProductionOrders(): Promise<ProductionOrder[]> {
     await delay(200 + Math.random() * 100);
     return mockOrders.map(order => validateProductionOrder(order));
   }
   ```

4. **Test that filter logic works with mock data:**
   ```typescript
   test('filter by Completed state returns only Completed orders', () => {
     const completed = mockOrders.filter(o => o.state === "Completed");
     expect(completed.length).toBeGreaterThan(0);
     expect(completed.every(o => o.state === "Completed")).toBe(true);
   });
   ```

5. **When real API arrives, add a DTO mapper:**
   ```typescript
   // Real API might have different structure
   interface RealAPIOrder {
     doc_id: string;
     doc_number: string;
     cust_name: string;
     prod_desc: string;
     wgt_lbs: number;
     dlv_time: string;
     prod_state: 'C' | 'M' | 'B' | 'P'; // Encoded states
     mill_code: '1' | '2' | '3'; // Encoded mill lines
   }

   const STATE_MAP: Record<RealAPIOrder['prod_state'], ProductionState> = {
     'C': 'Completed',
     'M': 'Mixing',
     'B': 'Blocked',
     'P': 'Pending',
   };

   const MILL_MAP: Record<RealAPIOrder['mill_code'], MillLine> = {
     '1': 'Premix',
     '2': 'Excel',
     '3': 'CGM',
   };

   function mapRealAPIToComponent(apiOrder: RealAPIOrder): ProductionOrder {
     return {
       id: apiOrder.doc_id,
       orderNumber: apiOrder.doc_number,
       customer: apiOrder.cust_name,
       product: apiOrder.prod_desc,
       weightLbs: apiOrder.wgt_lbs,
       deliveryTime: apiOrder.dlv_time,
       state: STATE_MAP[apiOrder.prod_state],
       millLine: MILL_MAP[apiOrder.mill_code],
     };
   }

   // Component only ever sees ProductionOrder, never raw API response
   ```

**Detection:**
- Filter returns 0 results for known data
- TypeScript doesn't catch mismatches (types match but values don't)
- Component breaks when real API data arrives

**When to address:**
Phase 2 (Data Service) — validate mock data structure matches component expectations before proceeding.

### Pitfall 4: useEffect Dependencies Causing Infinite Loops or Stale Closures in Filter Logic

**What goes wrong:**
You add a `useEffect` to sync filter results to a state variable. Filter dependency array includes objects/arrays that change every render, causing infinite loops. Or filter function uses stale closure of `activeStatuses` from an earlier render.

Example of the problem:
```typescript
// WRONG - infinite loop
const [filteredResults, setFilteredResults] = useState<ProductionOrder[]>([]);

useEffect(() => {
  const results = orders.filter(o => o.state === activeState);
  setFilteredResults(results);
}, [orders, activeState]); // OK so far

// But if you also do:
useEffect(() => {
  console.log('Filtered:', filteredResults); // filteredResults in deps = infinite
}, [orders, activeState, filteredResults]); // This triggers on every filter change
```

**Why it happens:**
React hooks require discipline with dependency arrays. Common mistakes:
1. Including derived state as dependency (effect that creates it depends on it)
2. Creating inline objects/functions in dependency array ([] and {} are new every render)
3. Forgetting to include dependencies, leading to stale closures in filter callbacks
4. Using `useEffect` for derived state when `useMemo` is correct

Your `OrdersTable` uses `useMemo` correctly, but extending to a new component tempts `useEffect` shortcuts.

**Consequences:**
- Component renders, effect runs, state updates, component re-renders, effect runs again → infinite loop
- Browser tab freezes or becomes unresponsive
- Or: filter callback uses wrong version of `activeStatuses`, filters don't work correctly
- Hard to debug (errors don't always propagate clearly, just feels slow)

**Prevention:**
1. **Use `useMemo` for derived state, not `useEffect`** (this is the main pattern):
   ```typescript
   // RIGHT - computed on every render, cached when deps change
   const filteredResults = useMemo(() => {
     return orders.filter(o =>
       activeStates.size === 0 || activeStates.has(o.state)
     );
   }, [orders, activeStates]);

   // WRONG - creates extra state and effect
   const [filteredResults, setFilteredResults] = useState<ProductionOrder[]>([]);
   useEffect(() => {
     setFilteredResults(
       orders.filter(o => activeStates.has(o.state))
     );
   }, [orders, activeStates]);
   ```

2. **Extract filter functions outside component** to simplify dependencies:
   ```typescript
   // Stable function, doesn't change between renders
   const filterByState = (orders: ProductionOrder[], states: Set<ProductionState>): ProductionOrder[] => {
     if (states.size === 0) return orders;
     return orders.filter(o => states.has(o.state));
   };

   // In component:
   const filteredResults = useMemo(() =>
     filterByState(orders, activeStates),
     [orders, activeStates]
   );
   ```

3. **Use ESLint plugin to catch dependency issues:**
   ```json
   {
     "eslintConfig": {
       "plugins": ["react-hooks"],
       "rules": {
         "react-hooks/exhaustive-deps": "error"
       }
     }
   }
   ```
   This catches missing or extra dependencies automatically.

4. **If you must use useEffect** (for side effects, not derived state):
   ```typescript
   // Correct: side effect with proper dependencies
   useEffect(() => {
     // This runs when filters change, not on every render
     logAnalytics('filters_changed', { states: Array.from(activeStates) });
   }, [activeStates]); // Specific dependency, not derived state
   ```

5. **Memoize objects used in dependency arrays:**
   ```typescript
   // Problem: filterOptions is new object every render
   const filterOptions = { states: activeStates, search: searchTerm };
   const filteredResults = useMemo(() =>
     filterByStateAndSearch(orders, filterOptions), // Depends on new object every render!
     [orders, filterOptions] // Infinite loop
   );

   // Solution: memoize the object
   const filterOptions = useMemo(() => ({
     states: activeStates,
     search: searchTerm,
   }), [activeStates, searchTerm]);

   const filteredResults = useMemo(() =>
     filterByStateAndSearch(orders, filterOptions),
     [orders, filterOptions]
   );
   ```

**Detection:**
- Component renders infinitely (browser slows down noticeably)
- Filter doesn't respond to state changes (uses stale value from closure)
- ESLint warnings about missing/extra dependencies
- Performance profiler shows constant re-renders

**When to address:**
Phase 1 (Filter Pills) — when implementing the filter logic itself. Get dependencies right from the start.

### Pitfall 5: Filter Behavior Changes When Data Service Changes Structure

**What goes wrong:**
You implement filters assuming `order.millLine === "Premix"` (exact string match). Later, real API returns `order.mill_code: "1"` (encoded). Filter logic breaks—component doesn't error, just returns no results.

Or: Mock data has `deliveryTime: "8:30 AM"` (11 orders in morning, 1 in afternoon). Real API has UTC timestamps. Filtering/sorting by time gives different results.

**Why it happens:**
Component filters and data service evolve independently. If service returns different field types, formats, or values, component filter logic breaks silently.

**Consequences:**
- Filter broken with no error message
- Swapping mock for real API requires re-implementing filters entirely
- Different filter logic exists in multiple places (tests, real code, different components)

**Prevention:**
(See Pitfall 3 above — this is the integration consequence)

Specifically for filters:
1. **Create a filter interface that service must satisfy:**
   ```typescript
   // Component cares about these properties for filtering
   interface Filterable {
     state: ProductionState;
     millLine: MillLine;
     customer: string;
   }

   // Service must return something compatible
   function validateFilterable(data: ProductionOrder): asserts data is ProductionOrder & Filterable {
     if (!VALID_STATES.includes(data.state)) throw new Error('Invalid state');
     if (!VALID_MILLS.includes(data.millLine)) throw new Error('Invalid mill');
     if (typeof data.customer !== 'string') throw new Error('Invalid customer');
   }
   ```

2. **Use DTO mapper with validation:**
   ```typescript
   // Real API structure
   interface APIProductionOrder {
     order_id: string;
     order_num: string;
     customer_name: string;
     product_desc: string;
     weight_lbs: number;
     delivery_time: string;
     state_code: string; // "C" | "M" | "B" | "P"
     mill_code: string;  // "1" | "2" | "3"
   }

   // Mapper validates and transforms
   function mapAndValidate(apiOrder: APIProductionOrder): ProductionOrder {
     const state = STATE_DECODE[apiOrder.state_code];
     if (!state) throw new Error(`Unknown state code: ${apiOrder.state_code}`);

     const millLine = MILL_DECODE[apiOrder.mill_code];
     if (!millLine) throw new Error(`Unknown mill code: ${apiOrder.mill_code}`);

     return {
       id: apiOrder.order_id,
       orderNumber: apiOrder.order_num,
       customer: apiOrder.customer_name,
       product: apiOrder.product_desc,
       weightLbs: apiOrder.weight_lbs,
       deliveryTime: apiOrder.delivery_time,
       state: state as ProductionState,
       millLine: millLine as MillLine,
     };
   }
   ```

3. **Add integration tests** that verify filter logic works:
   ```typescript
   test('can filter by state after mapping from API structure', () => {
     const apiOrder: APIProductionOrder = {
       order_id: '1',
       state_code: 'C',
       mill_code: '1',
       // ...
     };

     const mapped = mapAndValidate(apiOrder);
     const filtered = orders.filter(o => o.state === 'Completed');

     expect(filtered).toContain(mapped);
   });
   ```

**Detection:**
- Filter returns 0 results with known data
- Real API integration breaks filter logic
- Different components have different filter implementations (copy-paste problem)

**When to address:**
Phase 2 (Data Service) — validate mock data structure before wiring filters to it.

### Pitfall 6: Design Token Mismatches Between Figma and Code

**What goes wrong:**
Figma shows filter pill with rounded-xl (15px border radius) and gap-1.5 (6px) between icon and label. Implementation uses rounded-lg (12px) and gap-1 (4px). Small differences compound—UI looks "off" but you can't articulate why.

**Why it happens:**
Pixel-perfect matching requires discipline. Figma measurements and Tailwind/CSS values must map exactly. One wrong value cascades (if rounded-lg is wrong, all pills look different from design).

Your project has good spacing defaults, but when creating new components, easy to estimate instead of measure.

**Consequences:**
- UI doesn't match approved design
- Polish phase requires re-implementing components
- Visual inconsistency between designed and implemented state

**Prevention:**
1. **Create a design token extraction document** before coding:
   ```markdown
   ## Filter Pill Tokens (from mill-production.pen)

   ### Active State
   - bg-color: #4fd1c5 (primary)
   - text-color: #ffffff (white)
   - border-radius: 15px (rounded-xl)
   - padding: 8px 10px (py-2 px-2.5)
   - gap: 6px (gap-1.5)
   - shadow: 0 2px 4px rgba(0,0,0,0.08)

   ### Inactive State
   - bg-color: #f0f0f0 (gray-100)
   - text-color: #718096 (gray-600)
   - border-radius: 15px (rounded-xl)
   - padding: 8px 10px (py-2 px-2.5)
   ```

2. **Inspect in Figma/Zeplin, don't estimate:**
   - Don't guess spacing; use inspector tool
   - Don't assume color; use eyedropper
   - Copy values directly from design tool

3. **Test visual alignment in browser:**
   - Open design file in one window, browser dev tools in another
   - Use overlay tools (Pixel Perfect Pro, SuperDiff) to compare
   - Check every interactive state: default, hover, active, disabled

4. **Component code template with documented tokens:**
   ```typescript
   function FilterPill({ isActive, label, count }: FilterPillProps) {

     const bgClass = isActive ? 'bg-primary' : 'bg-gray-100';  // From design

   }
   ```
   All token values documented and sourced from design.

**Detection:**
- Visual regression: Screenshot before/after, compare side-by-side with design
- Pixel-by-pixel comparison: Do spacing/colors match exactly?
- Review checklist: Does every interactive state match design?

**When to address:**
Phase 3 (Design Polish) — after filter logic works, verify pixel-perfect matching before shipping.

### Pitfall 7: Empty Results State Not Designed or Implemented

**What goes wrong:**
Filter logic works great with full dataset. User applies a filter (e.g., "show only Completed orders") that matches 0 orders. Component shows blank white space. User doesn't know if loading, if filter is broken, or if there genuinely are no results.

Your `OrdersTable` has a good empty state message with "Clear filters" button. Mill production view needs the same thoughtfulness.

**Why it happens:**
Happy path testing is quick (full data), edge cases (empty results) feel less important. Not designing empty states upfront—they're bolted on as afterthought.

**Consequences:**
- Users confused (is the UI broken or are there no results?)
- No way to recover except manual filter reset
- Looks unprofessional

**Prevention:**
1. **Design empty states in Figma** (not just code them):
   - "No orders in this state" with icon
   - "No results matching your filter"
   - "All filters cleared" (after user clicks clear)

2. **Create test datasets for edge cases:**
   ```typescript
   const testCases = [
     { name: 'empty', orders: [] },
     { name: 'single', orders: [mockOrders[0]] },
     { name: 'all-completed', orders: mockOrders.filter(o => o.state === 'Completed') },
     { name: 'no-premix', orders: mockOrders.filter(o => o.millLine !== 'Premix') },
   ];
   ```

3. **Empty state checklist:**
   - Message is visible and clear ("No Completed orders")
   - Layout doesn't shift (use min-height for consistent spacing)
   - "Clear filters" button works and resets all state
   - Keyboard navigation doesn't error (no crash if no rows selected)
   - Icon/illustration matches design
   - Text color uses text-secondary (not too dark)

4. **Implement empty state component:**
   ```typescript
   function EmptyState() {
     return (
       <div className="flex flex-col items-center justify-center py-12 text-center">
         <div className="h-12 w-12 rounded-full bg-gray-100 mb-4" />
         <p className="text-text-primary text-sm font-medium">No orders in this state</p>

         <button
           onClick={clearAllFilters}
           className="text-primary mt-4 text-sm hover:underline"
         >
           Clear all filters
         </button>
       </div>
     );
   }
   ```

**Detection:**
- Apply filter that matches 0 orders, check for empty state message
- Visual check: Is empty state designed and matches design system?

**When to address:**
Phase 1 (Filter Pills) — implement empty state alongside filter logic.

### Pitfall 8: Loading State Confusion When Filters and Data Fetch Mix

**What goes wrong:**
Mock service has 200ms delay to simulate network latency. User sees loading skeleton, applies filters while loading, component tries to filter undefined data, or shows loading state without indicating which filters are active.

Or: Component shows "Loading..." but doesn't disable filters. User applies filters during load, state changes, data arrives with different structure, filters apply incorrectly.

**Why it happens:**
Loading state handling often added last ("just show spinner while fetching"). Combining loading state + active filters requires careful UX thinking.

**Consequences:**
- Race condition: filter changes while data loading, stale data applied
- User confusion: can't tell if filters are active during load
- Filters disable unexpectedly (can't apply new filters during load)

**Prevention:**
1. **Show active filters during loading:**
   ```typescript
   {loading ? (
     <div>
       <p className="text-sm text-text-secondary">
         Loading {activeStatuses.size > 0 ? 'filtered results' : 'all orders'}...
       </p>
       <LoadingSkeleton />
     </div>
   ) : (
     <div>
       {/* Results */}
     </div>
   )}
   ```

2. **Disable filters during loading** (prevent race conditions):
   ```typescript
   <button
     onClick={() => toggleState("Completed")}
     disabled={loading}
     className={`transition-opacity ${loading ? 'opacity-50 cursor-not-allowed' : 'cursor-pointer'}`}
   >
     Completed
   </button>
   ```

3. **Debounce filter changes to avoid rapid toggling:**
   ```typescript
   const [pendingFilters, setPendingFilters] = useState<Set<ProductionState>>(new Set());

   const toggleState = (state: ProductionState) => {
     setPendingFilters(prev => {
   ```

### Pitfall 9: Filter Pill Spacing/Alignment Issues on Different Viewports

**What goes wrong:**
Filter pills look perfectly aligned in Figma (1920px viewport) but wrap to two lines on 1440px. Or pills don't fit in container, overflow with horizontal scroll appearing.

**Why it happens:**
Horizontal layout (flexbox `flex-row` with `gap-2.5`) depends on container width. Estimating width on design is different from real breakpoints. Pills don't wrap in designer's wide screen but do on standard laptop.

**Consequences:**
- Layout shift on different screens (looks janky)
- Pills overflow unexpectedly
- Responsive behavior not thought through

**Prevention:**
1. **Test on multiple viewport sizes:**
   - 1920px (desktop)
   - 1440px (laptop)
   - 1024px (tablet landscape)
   - 768px (tablet portrait)
   - 375px (mobile)

   Use Chrome DevTools device emulation.

2. **Set responsive constraints:**
   ```typescript
   // Pills wrap on small screens, stay horizontal on large
   <div className="flex flex-wrap gap-2.5 md:flex-nowrap lg:flex-nowrap">
     {/* Filter pills */}
   </div>
   ```

3. **Consider overflow handling:**
   ```typescript
   // Horizontal scroll if needed
   <div className="overflow-x-auto">
     <div className="flex gap-2.5 flex-shrink-0">
       {/* Pills won't shrink, scroll if needed */}
     </div>
   </div>
   ```

4. **Test actual layout at deployment viewport:**
   - Your dashboard probably targets specific viewport range
   - Test at that range, don't over-engineer for mobile if not needed

### Pitfall 10: Debounce Timing Wrong for Search Filter

**What goes wrong:**
Search filter has 300ms debounce (your current implementation), but on slow connection, waiting 300ms + network latency feels slow. Or: no debounce, filters on every keystroke, kills performance.

**Why it happens:**
Debounce is a balance: responsiveness (user feels snappy) vs efficiency (don't filter constantly). Right timing depends on dataset size and calculation complexity.

**Consequences:**
- Too slow: UI feels laggy ("I typed 2 seconds ago, why isn't it filtered yet?")
- Too fast: Performance issues with large datasets
- Inconsistent: Different data sizes need different timing

**Prevention:**
1. **Measure filter calculation time:**
   ```typescript
   const filteredResults = useMemo(() => {
     const start = performance.now();
     const result = orders.filter(o => matchesSearch(o, searchTerm));
     const time = performance.now() - start;
     if (time > 10) console.log(`Filter: ${time.toFixed(2)}ms`);
     return result;
   }, [orders, searchTerm]);
   ```

   - <10ms: no debounce needed
   - 10-50ms: 100-200ms debounce
   - >50ms: 300-500ms debounce

2. **Test with realistic data volume:**
   - Current mock: ~12 orders (fast, <1ms)
   - Real production: could be 500+ orders (slower, 10-50ms)
   - Adjust debounce accordingly

3. **Your current `useDebounce` hook is correct:**
   ```typescript
   export function useDebounce<T>(value: T, delay: number): T {
     const [debouncedValue, setDebouncedValue] = useState(value);

     useEffect(() => {
       const handler = setTimeout(() => setDebouncedValue(value), delay);
       return () => clearTimeout(handler);
     }, [value, delay]);

     return debouncedValue;
   }

   // Usage:
   const debouncedSearch = useDebounce(searchTerm, 300);
   const filteredResults = useMemo(() =>
     orders.filter(o => matchesSearch(o, debouncedSearch)),
     [orders, debouncedSearch]
   );
   ```

**Detection:**
- Type in search, measure perceived lag
- Console: log filter calculation time
- Performance profiler: check re-render frequency

**When to address:**
Phase 2 (Data Service) — when wiring to larger dataset.

### Pitfall 12: Performance Degradation with Larger Datasets

**What goes wrong:**
Mock data works fine with 12 orders. Real data has 500+ orders. Filtering becomes slow (noticeable lag), UI stutters when applying multiple filters.

**Why it happens:**
Current filter implementation loops over array multiple times (once per filter dimension). With 12 items: <1ms. With 500 items: 5-10ms per filter. Not terrible, but noticeable.

**Consequences:**
- Dashboard feels slow with real data
- Users lose confidence in UI responsiveness
- Keyboard navigation stutters

**Prevention:**
1. **Combine filters into single pass:**
   ```typescript
   // SLOW: 3 filter iterations
   let result = orders.filter(o => activeStatuses.has(o.state));
   result = result.filter(o => o.millLine === selectedMill);
   result = result.filter(o => matchesSearch(o, searchTerm));

   // FAST: 1 filter iteration
   const result = orders.filter(o =>
     (activeStatuses.size === 0 || activeStatuses.has(o.state)) &&
     (!selectedMill || o.millLine === selectedMill) &&
     matchesSearch(o, searchTerm)
   );
   ```

2. **Measure performance with large mock dataset:**
   ```typescript
   // Generate 500 mock orders for testing
   const largeDataset = Array.from({ length: 500 }, (_, i) =>
     createMockOrder({ orderNumber: `ORD-${260000 + i}` })
   );

   // Measure filter time
   const start = performance.now();
   const filtered = largeDataset.filter(o => activeStatuses.has(o.state));
   console.log(`Filtered 500 items: ${(performance.now() - start).toFixed(2)}ms`);
   ```

3. **If >100ms, virtualize the list** (render only visible rows):
   ```typescript
   import { FixedSizeList } from 'react-window';

   <FixedSizeList
     height={600}
     itemCount={filteredResults.length}
     itemSize={50}
     width="100%"
   >
     {({ index, style }) => (
       <div style={style}>
         <ProductionCard order={filteredResults[index]} />
       </div>
     )}
   </FixedSizeList>
   ```

**Detection:**
- Load larger dataset (or generate mock dataset with 500+ items)
- Apply filters, measure lag
- Profiler: check React render time, filter calculation time

**When to address:**
Phase 2 (Data Service) — if integrating large dataset. Otherwise defer to v1.2.

## Executive Summary

v1.1 scales an existing working component (mill production view with static data) by adding three new capabilities:

1. **Filter Pills** — Status-based filtering with count badges (proven pattern exists in OrdersTable)
2. **Design Polish** — Pixel-perfect matching to approved .pen design using design token system
3. **Data Service** — Mock JSON service with realistic structure from Book1.xlsx data

The primary risk is **complexity compounding**. Each layer introduces failure modes that interact:
- Filter logic depends on data structure (Pitfall 3)
- Design tokens bypass leads to maintenance nightmare (Pitfall 2)
- Count calculation subtle bugs persist through design phase (Pitfall 1)

**The good news:** Your OrdersTable component provides a working template for filter architecture. You've already made the right choices (useMemo for derived state, URL-based filter design decision deferred). The work is replicating those patterns carefully, not inventing new ones.

## Key Findings

### Technology & Architecture

**Filter Architecture (from existing OrdersTable):**
- Use `useMemo` for derived state (filter calculations), NOT `useEffect`
- Separate count calculations with asymmetrical filtering (don't self-reference)
- Debounce search input (300ms current implementation works)
- Keep filter state local to component (URL state deferred for v1.1)

**Mock Data Service:**
- Current implementation correct pattern (async function returning Promise, simulated delay)
- **Critical:** Mock data type must match component filter expectations exactly
- Need runtime validation to catch type/format mismatches before shipping
- DTO mapper pattern ready for real API swap

**Design System:**
- Strong token system already in place (globals.css, Tailwind theme)
- **Critical:** Never bypass tokens with hardcoded colors or spacing
- Use Tailwind utilities exclusively (px-, gap-, bg-, text- classes)
- For custom values: extend config, add to tokens, use via classes

### Filter State Management Pattern

Your OrdersTable implements this correctly:

```typescript
// 1. Derived state (not useState)
const filteredOrders = useMemo(() => {
  let result = orders;
  if (activeStatuses.size > 0) {
    result = result.filter(o => activeStatuses.has(o.status));
  }
  if (searchTerm) {
    result = result.filter(o => matchesSearch(o, searchTerm));
  }
  return result;
}, [orders, activeStatuses, searchTerm]);

// 2. Count calculation (self-referential filter excluded)
const statusCounts = useMemo(() => {
  let toCount = orders;
  if (searchTerm) { // Include search filter
    toCount = toCount.filter(o => matchesSearch(o, searchTerm));
  }
  // DON'T filter by activeStatuses here (self-referential)
  return toCount.reduce((acc, o) => {
    acc[o.status] = (acc[o.status] || 0) + 1;
    return acc;
  }, {});
}, [orders, searchTerm]);
```

**For mill production view, this pattern adapts to:**
- Filter dimensions: state (Completed/Mixing/Blocked/Pending), mill line (Premix/Excel/CGM), and optionally texture/customer
- Count logic: must exclude self-referential filter from count calculation
- Empty state: when 0 results, show "No orders in [filter name]" message

## Implications for Roadmap

### Phase 1: Design Filter Pills

**Scope:** Implement filter UI with correct interaction and counts

**Must Address (Before Moving to Phase 2):**

1. **Filter count matrix documented** (prevents silent bugs)
   - Which filters contribute to which count calculations
   - What gets excluded from each count (self-referential rule)
   - Test matrix covering all combinations

2. **Implementation pattern locked down**
   - useMemo for filter calculations (NOT useEffect)
   - Dependency array complete (ESLint rule enabled)
   - No infinite loops possible

3. **Empty state implemented alongside filter logic**
   - "No [state] orders" message when 0 results
   - "Clear filters" button that works
   - Keyboard navigation doesn't break

4. **Design tokens verified**
   - All colors reference CSS classes (no hardcoded hex)
   - All spacing uses Tailwind utilities
   - Hover and active states match design

**Why Order Matters:**
If filter counts are wrong, Phase 3 (Design Polish) will discover it during visual testing. Fixing at that point requires rework. Test counts thoroughly in Phase 1.

### Phase 2: Wire Data Service

**Scope:** Connect mock service to filters, validate everything works together

**Must Address (Before Moving to Phase 3):**

1. **Mock data validated against interface**
   - Runtime validation catches type mismatches
   - Factory function ensures all fields present
   - Integration test: filters work with service data

2. **Loading/error states finalized**
   - Loading state doesn't disable filters (or does—decide and lock)
   - Error recovery path exists
   - Active filters shown during load

3. **Performance baseline established**
   - Measure filter calculation time with current mock data
   - Establish acceptable debounce timing
   - Plan for scale if real data is 10x larger

4. **DTO mapper prepared**
   - Real API structure documented (or guess what it might be)
   - Mapper function ready to transform API response to ProductionOrder
   - Mapped data tested with filter logic

**Why Order Matters:**
If mock data structure doesn't match component expectations, Phase 3 design polish creates mismatch. Filters that work in Phase 2 might break when real API arrives. Test integration thoroughly.

### Phase 3: Design Polish

**Scope:** Pixel-perfect matching to .pen file, responsive testing

**Can Assume (Already Locked Down):**
- Filter logic correct (Phase 1)
- Data service working (Phase 2)
- Component structure stable

**Must Address:**

1. **Design token extraction from .pen**
   - Extract exact colors, spacing, shadows for each state
   - Map to Tailwind/CSS tokens
   - Document for code review

2. **Pixel-perfect visual matching**
   - Use design overlay tools (Pixel Perfect Pro, etc.)
   - Check every pixel: colors, spacing, border radius
   - All interactive states match design

3. **Responsive testing**
   - Test on target viewport range (probably desktop-focused for operations dashboard)
   - Pills don't overflow or wrap unexpectedly
   - Layout stable across viewport sizes

4. **Visual regression testing**
   - Screenshot before/after
   - Compare with design side-by-side
   - QA checklist covers all states

**Why Order Matters:**
Polish phase assumes prior phases are stable. Don't start polish work until filter logic and data service confirmed working.

## Critical Pitfalls to Prevent

| Pitfall | Phase | Prevention | Cost if Missed |
|---------|-------|-----------|-----------------|
| **Filter counts wrong** (Pitfall 1) | Phase 1 | Document matrix, test all combinations | Rework in Phase 3, looks broken |
| **Mock data mismatches** (Pitfall 3) | Phase 2 | Runtime validation, factory function | Integration fail when real API arrives |
| **Design tokens bypassed** (Pitfall 2) | Phase 3 | ESLint rule, code review, never hardcode colors | Maintenance nightmare, every design change requires hunt-and-replace |
| **useEffect infinite loops** (Pitfall 4) | Phase 1 | Use useMemo, enable ESLint, validate dependencies | Browser freeze, hard to debug |
| **Empty state missing** (Pitfall 7) | Phase 1 | Design in Figma, implement with filters | UX confusion when 0 results |

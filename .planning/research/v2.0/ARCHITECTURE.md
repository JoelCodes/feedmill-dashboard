## Decision 1: Data Layer Organization

### Directory layout

Follow the Drizzle canonical pattern confirmed in official docs (drizzle-team/drizzle-orm-docs):

```
src/db/schema/orders.ts     — pgTable definitions for production_orders
src/db/schema/events.ts     — pgTable definitions for order_events
src/db/schema/index.ts      — re-exports all tables (consumed by drizzle.config.ts)
src/db/queries/orders.ts    — typed query functions
src/db/queries/events.ts    — typed query functions
src/db/index.ts             — db singleton
drizzle/migrations/         — SQL migration files from drizzle-kit generate
drizzle.config.ts           — at repo root, pointing at src/db/schema/index.ts
```

`drizzle.config.ts` stays at repo root (Drizzle Kit convention). Migration SQL files go in `drizzle/migrations/` (separate from `src/` so they are not bundled into the Next.js build).

### Do RSCs call queries directly or via a service wrapper?

**Use a thin query function layer in `src/db/queries/`; RSCs call those functions directly.**

Rationale: The existing pattern is `page.tsx` calling `await getProductionOrders()` from `src/services/millProduction.ts`. The production page will call `await getProductionOrders()` from `src/db/queries/orders.ts` — same call site shape, different import. A service wrapper on top of queries adds a layer with no benefit for a single-tenant, single-mill app at this scale.

Query functions are not "use server" — they are plain async functions importable only from server-side code. This is consistent with the existing service pattern and the security-patterns.md rule: service/query modules are never imported from `'use client'` files.

### What happens to `mockOrdersService.ts` / `millProduction.ts`?

**Keep `src/services/millProduction.ts` as the mock/test alternative. Do not delete it.**

Rationale:
- `src/app/demo/mill-production/page.tsx` will continue to call `getProductionOrders()` from the mock service. Demo namespace is explicitly out of scope for v2.0.
- Jest tests for demo pages mock this module. Deleting it breaks the test suite.
- If Storybook is added later, mock services provide deterministic fixtures.
- The naming divergence (`src/services/millProduction.ts` vs `src/db/queries/orders.ts`) is intentional — it makes the mock vs. real distinction visible at the import path level.

## Decision 2: Mutation Pattern — Server Actions

**Use server actions (`'use server'` in `src/actions/transitions.ts`) for all status transitions.**

Rationale vs. route handlers:

| Criterion | Server Actions | Route Handler |
|---|---|---|
| RSC cache invalidation | `revalidatePath('/')` in the action body; RSC re-fetches and re-renders automatically | Requires separate `router.refresh()` call from client after fetch |
| Error propagation | Returned error object surfaced directly to the client component | Must parse HTTP response body |
| Progressive enhancement | Works with plain HTML `<form>` before JS hydrates | Requires JS |
| Type safety | Action is a typed function; client gets return type inference | HTTP response body needs manual typing |
| External access | Not needed — transitions are internal | Route handler needed only for webhooks / external callers |
| Vercel body limit | Actions have a 1 MB default limit (configurable via `next.config.ts`); transitions carry only an order ID + new state (< 1 KB) | N/A |

Status transitions carry a tiny payload (orderId + targetState), so the 1 MB default limit is irrelevant. `revalidatePath('/')` after a transition means the RSC re-renders with fresh DB data without a client-side polling cycle.

Confirmed pattern from Drizzle official docs tutorial (drizzle-nextjs-neon):

```typescript
// src/actions/transitions.ts
"use server";
import { eq } from "drizzle-orm";
import { revalidatePath } from "next/cache";
import { db } from "@/db";
import { productionOrders } from "@/db/schema/orders";
import { orderEvents } from "@/db/schema/events";
import { auth } from "@clerk/nextjs/server";
import type { ProductionState } from "@/types/millProduction";

export async function transitionOrderState(
  orderId: string,
  newState: ProductionState
): Promise<{ success: true } | { error: string }> {
  const { userId } = await auth();
  if (!userId) return { error: "Unauthenticated" };
  // ... update production_orders, insert order_events row
  revalidatePath("/");
  return { success: true };
}
```

## Decision 3: File Upload (XLSX Import)

### Vercel constraints (verified from official docs, 2026-02-24)

- **Request body limit: 4.5 MB** — hard limit on Hobby and Pro. Returns HTTP 413 if exceeded.
- **Server action default: 1 MB** — configurable in `next.config.ts` up to the 4.5 MB function limit.
- **Function duration (Hobby/Pro with Fluid Compute): 300 s default** — sufficient for parsing.

### Realistic XLSX size for a feed mill order sheet

The example `Book1.xlsx` in `example-data/` has 33 rows and 8 columns. A full day's production schedule at a single mill is typically 100–500 rows. At 8 columns of text/numeric data, a raw XLSX of 500 rows compresses to approximately 20–50 KB — well under the 4.5 MB limit with margin for thousands of rows. A realistic upper bound for a single-mill daily import is under 500 KB.

**Conclusion: the 4.5 MB body limit is not a practical constraint for this use case.** The workaround path is not needed for v2.0.

### Recommended import flow

**Use a server action for XLSX import** (under 4.5 MB path):

1. Client renders a file input inside a form that `action={importXlsxAction}`.
2. `importXlsxAction` in `src/actions/import.ts` receives the `FormData`, extracts the `File` object, reads it as an `ArrayBuffer`, passes it to `xlsx.read()` (the `xlsx` / SheetJS library), maps rows to `ProductionOrder` inserts, writes to `production_orders` in a single batch `db.insert(...).values(rows)`, calls `revalidatePath('/')`.
3. Return `{ imported: N }` or `{ error: string }`.

The action-based path avoids a separate route handler and keeps the import fully integrated with RSC revalidation. `xlsx` (SheetJS) parses `.xlsx`, `.xls`, and `.csv` server-side with no browser dependency.

**If an import ever exceeds 4.5 MB** (e.g., quarterly bulk loads): use Vercel Blob client upload. The `@vercel/blob` package provides `handleUpload` in a route handler at `src/app/api/mill-production/import/route.ts` that issues a presigned URL; the client uploads directly to Blob storage bypassing the function; the server receives an `onUploadCompleted` callback, reads the file from Blob, parses, and inserts. This path is deferred until demonstrated need.

## Decision 4: Real-Time Mechanism

**Recommendation: polling via `router.refresh()` on a fixed interval for v2.0.**

### Tradeoff analysis

| Pattern | Complexity | Vercel fit | Multi-tab | Accuracy |
|---|---|---|---|---|
| `router.refresh()` polling (30 s) | Low — custom hook, no new infra | Excellent — stateless HTTP, no streaming concerns | Each tab polls independently (no cross-tab push) | ±30 s lag |
| SSE via route handler | Medium — route handler + ReadableStream + `force-dynamic`; Hobby 10 s timeout concern (mitigated by Fluid Compute 300 s) | Good with `force-dynamic` and event IDs; Fluid Compute changes the calculus | Yes — all tabs connected to SSE receive updates | Near-instant |
| Pusher/Ably | High — third-party dependency, env vars, SDK on client | Good | Excellent | Near-instant |

**Why polling wins for v2.0:**

- The production page is a physical dashboard viewed by one operator per mill line. Sub-second latency is not required. 30-second polling fits operational rhythm (mixing cycles are 5–15 minutes).
- `router.refresh()` is a single line in a `useEffect` with `setInterval`. Zero new infrastructure.
- Calling `router.refresh()` in a client component triggers RSC re-render on the server, re-fetching from the DB, exactly like a full navigation — data is always current at the moment of the refresh.
- SSE on Vercel with Fluid Compute (300 s default) now works for single-connection use cases, but still requires `force-dynamic` on the route, a persistent connection held open, and EventSource reconnect handling. That is disproportionate for a 30-second-acceptable refresh cycle.
- External services (Pusher/Ably) introduce a new vendor dependency that is unnecessary at this scale.

**Upgrade path**: if operators need sub-10-second updates (e.g., blocked-order alerts), swap to SSE by adding a `src/app/api/mill-production/events/route.ts` with `export const dynamic = 'force-dynamic'`. The polling hook in the client component is replaced by an `EventSource` listener that calls `router.refresh()` on each received event. No schema changes required.

### Polling implementation shape

```typescript
// src/components/MillProductionProUI.tsx  (client component)
'use client';
import { useRouter } from 'next/navigation';
import { useEffect } from 'react';

const REFRESH_INTERVAL_MS = 30_000;

export function MillProductionProUI({ orders }: { orders: ProductionOrder[] }) {
  const router = useRouter();
  useEffect(() => {
    const id = setInterval(() => router.refresh(), REFRESH_INTERVAL_MS);
    return () => clearInterval(id);
  }, [router]);
  // render orders ...
}
```

## Decision 5: URL State for Filters

### Pattern

`page.tsx` (RSC) receives `searchParams` as a prop. Pass filter values from `searchParams` to the Drizzle query. Pass the same values down to the client component as initial state.

The client component updates the URL via `useRouter().push` with new search params; the RSC re-renders with the new params via normal Next.js navigation.

```typescript
// src/app/page.tsx
export default async function ProductionPage({
  searchParams,
}: {
  searchParams: Promise<Record<string, string | string[]>>;
}) {
  await requireRole('mill_operator');
  const params = await searchParams; // Next.js 16 — await required
  
  const millLine = params.millLine as MillLine | undefined;
  const states = Array.isArray(params.state)
    ? (params.state as ProductionState[])
    : params.state
    ? [params.state as ProductionState]
    : undefined;

  const orders = await getProductionOrders({ millLine, states });
  
  return (
    <DashboardLayout>
      <Suspense fallback={<MillProductionSkeleton />}>
        <MillProductionProUI orders={orders} millLine={millLine} states={states} />
      </Suspense>
    </DashboardLayout>
  );
}
```

The `Suspense` wrapper is required because `MillProductionProUI` will use `useSearchParams()` for reading current filter state client-side (Next.js 16 requires `useSearchParams` to be inside a Suspense boundary — consistent with the existing `OrdersTableContent` pattern in `src/app/demo/orders/page.tsx`).

### Drizzle query with optional filters

```typescript
// src/db/queries/orders.ts
import { and, eq, inArray } from 'drizzle-orm';
import { db } from '@/db';
import { productionOrders } from '@/db/schema/orders';

export async function getProductionOrders(filters?: {
  millLine?: MillLine;
  states?: ProductionState[];
}) {
  const conditions = [];
  if (filters?.millLine) conditions.push(eq(productionOrders.millLine, filters.millLine));
  if (filters?.states?.length) conditions.push(inArray(productionOrders.state, filters.states));
  
  return db.select().from(productionOrders)
    .where(conditions.length ? and(...conditions) : undefined)
    .orderBy(productionOrders.deliveryTime);
}
```

**Do not use `nuqs`** for v2.0. The RSC-first architecture already handles URL state without a client library. Nuqs is valuable when you need optimistic URL updates with type coercion across many parameters — premature for a 3-filter use case.

## Decision 6: Role Gating for `/`

### Recommended role: `'mill_operator'`

Rationale:
- The production page at `/` is the first real feature. It is distinct from `demo` (which is for evaluating the mock product) and `user` (which is too generic).
- `mill_operator` communicates intent and is extensible: a supervisor might later get `mill_supervisor`, a manager gets `manager`, etc. Flat roles are maintained (no nested hierarchy) per the existing architecture decision.
- Using the `demo` role for production would conflate two different user populations (evaluators vs. operators), making it impossible to later restrict demo access without affecting real operators.
- Reusing `user` leaves no room to gate — all authenticated users would see the production page without role assignment, which is the wrong default for an ops dashboard.

### Type change

In `src/types/clerk.d.ts`, extend the `Role` union:

```typescript
export type Role = 'demo' | 'admin' | 'user' | 'mill_operator';
```

### Middleware change

The current middleware only gates `/demo/*`. The new `/` production page uses `requireRole('mill_operator')` at the RSC level (inner guard pattern from `docs/security-patterns.md` §2). No middleware change is strictly required — `auth.protect()` already ensures authentication, and `requireRole` handles the role check at the page boundary.

**Optional but recommended**: add a middleware-level coarse gate for production routes so unauthenticated requests fail fast at the Edge without invoking the Node.js function:

```typescript
// src/middleware.ts — addition
const isProductionRoute = createRouteMatcher(['/']);

// inside clerkMiddleware handler:
if (isProductionRoute(request)) {
  const { sessionClaims } = await auth();
  if (sessionClaims?.metadata?.role !== 'mill_operator') {
    return NextResponse.redirect(new URL('/sign-in', request.url));
  }
}
```

This is optional for v2.0 because the page-level `requireRole` is the canonical security enforcement point. The middleware gate is defense-in-depth.

### Impact on `docs/security-patterns.md`

Per §6 of that doc (onboarding checklist), a new row must be added to the §1 audit findings table when `/` is promoted to a role-gated page.

## Decision 7: Audit Trail Storage

**Use a separate `order_events` table (append-only log pattern).**

Rationale vs. alternatives:

| Option | Pros | Cons |
|---|---|---|
| Separate `order_events` table | Queryable history; supports timeline UI; append-only (no UPDATEs on log rows); scales cleanly | One extra query on detail view |
| JSON column on `production_orders` | Simpler schema | JSON not indexed; history not queryable with SQL; mutation requires read-modify-write |
| In-row `last_changed_at` + `last_changed_by` | Zero extra table | No history; loses prior states |

The existing `OrderDetails` component already renders a timeline of events. The `order_events` table maps directly to that UI pattern.

### Schema shape

```typescript
// src/db/schema/events.ts
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core';
import { productionOrders } from './orders';

export const orderEvents = pgTable('order_events', {
  id: uuid('id').defaultRandom().primaryKey(),
  orderId: uuid('order_id').notNull().references(() => productionOrders.id),
  fromState: text('from_state'),       // null on initial import
  toState: text('to_state').notNull(),
  changedBy: text('changed_by').notNull(),   // Clerk userId
  changedAt: timestamp('changed_at', { withTimezone: true }).notNull().defaultNow(),
  note: text('note'),                  // optional free-text (blocked reason, etc.)
});
```

**No soft deletes.** Events are insert-only. `productionOrders` records are updated in-place for `state`; events table records the transition history. This matches the existing `ProductionOrder` type's `state` field.

## Decision 8: Suggested Build Order

The dependency graph for v2.0 has four layers. Each layer must be complete before the next can start.

### Layer 0 — Role infrastructure (blocks everything)

1. Extend `Role` type in `src/types/clerk.d.ts` to add `'mill_operator'`.
2. Assign `mill_operator` role to test user(s) in Clerk Dashboard.
3. Add `mill_operator` test fixture to `src/test/fixtures/clerkAuth.ts`.
4. Optionally: add middleware coarse gate for production route.

**Why first:** Every RSC page guard and every action auth check depends on the role string being defined. The Jest fixture must exist before page tests can be written.

### Layer 1 — Schema and migration (blocks query functions and UI)

1. Install `drizzle-orm`, `drizzle-kit`, `@neondatabase/serverless` (or `@vercel/postgres`), `dotenv`.
2. Write `drizzle.config.ts` at repo root.
3. Write `src/db/schema/orders.ts` (`production_orders` table).
4. Write `src/db/schema/events.ts` (`order_events` table).
5. Write `src/db/schema/index.ts` (re-export).
6. Write `src/db/index.ts` (db singleton).
7. Run `npx drizzle-kit generate` → SQL migration in `drizzle/migrations/`.
8. Run `npx drizzle-kit migrate` against development DB.

**Why second:** Query functions import table definitions. Server actions import query functions. RSC page imports query functions. Nothing works before the schema and migrations exist.

### Layer 2 — Query functions and server actions (blocks UI)

1. Write `src/db/queries/orders.ts` (getProductionOrders with filter params; upsertOrder for import).
2. Write `src/db/queries/events.ts` (insertEvent; getEventsByOrderId).
3. Write `src/actions/transitions.ts` ("use server"; transitionOrderState; auth check; event insert; revalidatePath).
4. Write `src/actions/import.ts` ("use server"; XLSX parse; batch insert; revalidatePath).
5. Install `xlsx` (SheetJS) for XLSX parsing.

**Why third:** The UI components bind to action signatures. If the action API changes, UI bindings must be updated. Write actions before UI to avoid rework.

### Layer 3 — UI and page promotion (depends on all above)

1. Write `src/components/MillProductionProUI.tsx` (client wrapper; polling hook; filter pills; status transition buttons calling `transitionOrderState`; import form calling `importXlsxAction`).
2. Replace `src/app/page.tsx` (Coming Soon → async RSC; `requireRole('mill_operator')`; `searchParams` → Drizzle query; render `MillProductionProUI`).
3. Update `src/components/Sidebar.tsx` production nav links.
4. Update `docs/security-patterns.md` §1 audit table.

### Layer 4 — KPI cards (depends on Layer 3 data)

KPI cards (deferred since v1.0) can now be computed from real DB data:
- Total weight today (sum of `weight_lbs` for `state = 'Completed'`)
- Orders pending / mixing / blocked counts
- Throughput vs. target

These are derived from the same `getProductionOrders()` call — no additional queries needed if computed in the RSC before handing to the UI.

## Vercel Runtime Constraints Summary

| Feature | Constraint | v2.0 Impact |
|---|---|---|
| Request body size | 4.5 MB hard limit (Hobby + Pro) | XLSX import is safe under this limit for daily mill schedules (< 500 KB typical) |
| Server action body | 1 MB default; configurable up to 4.5 MB via `next.config.ts` `serverActions.bodySizeLimit` | Set to `'2mb'` in `next.config.ts` as a precaution; no presigned-URL flow needed for v2.0 |
| SSE function duration | Fluid Compute default 300 s (Hobby + Pro) | Polling preferred for v2.0; SSE feasible if needed in a later phase |
| Edge vs Node runtime | Drizzle + Neon serverless driver runs in Node.js runtime | RSC pages and API routes must NOT use `export const runtime = 'edge'` |
| DB connections | Serverless connection pooling required | Use `@neondatabase/serverless` with the `neon()` HTTP driver, or Neon's connection pooler endpoint — avoids TCP connection exhaustion from cold-start spikes |

## Migration Path: Mock → Real Service

The existing `src/app/demo/mill-production/page.tsx` imports from `src/services/millProduction.ts`. The new `src/app/page.tsx` imports from `src/db/queries/orders.ts`. These are two separate import paths pointing to two separate implementations — no aliasing or conditional branching needed.

The function signature contract is maintained:

| Mock (existing) | Real (new) |
|---|---|
| `getProductionOrders(): Promise<ProductionOrder[]>` | `getProductionOrders(filters?): Promise<ProductionOrder[]>` |
| `getOrdersByMillLine(millLine): Promise<ProductionOrder[]>` | covered by `getProductionOrders({ millLine })` |

Both return the same `ProductionOrder` type from `src/types/millProduction.ts`. The client component `MillProductionProUI` (or `MillProductionUI` for demo) receives `orders: ProductionOrder[]` as a prop — the prop interface does not change.

## Open Questions for Phase Research

1. **Neon vs. Vercel Postgres**: Both are Postgres-on-Vercel. Neon's serverless driver (`@neondatabase/serverless`) uses HTTP, which avoids TCP connection pooling problems in serverless. Vercel Postgres uses `@vercel/postgres`, which wraps Neon. Preference: use `@neondatabase/serverless` directly for more control over the HTTP driver. Confirm provider in DB setup phase.

2. **`next.config.ts` `serverActions.bodySizeLimit`**: Confirm the exact config key for Next.js 16. In Next.js 14+ it is `experimental.serverActions.bodySizeLimit` (or top-level in later versions). Check during install phase.

3. **Clerk JWT template update**: Adding `mill_operator` to the `Role` type is a code change. The Clerk Dashboard JWT template (`"role": "{{user.public_metadata.role}}"`) does not need changing — it already forwards any string in `publicMetadata.role`. Confirm this during role setup phase.

4. **`import_batches` table**: Consider adding an `import_batches` table to record each XLSX upload (file name, row count, imported_at, imported_by) for operational visibility. Not required for v2.0 MVP but cheap to add with the initial schema.
